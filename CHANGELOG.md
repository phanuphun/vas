### แก้ `parse_gsettings_uint()` อ่านค่า screen blank ผิดช่วง session/D-Bus เพิ่งเริ่มใหม่ๆ (race condition)
- **ที่มา**: ต่อจาก entry ด้านล่าง (monitors.xml sync-on-apply) — ผู้ใช้รายงานว่าหน้า `/display` โชว์ "Screen Blank: 3 นาที" ทั้งที่ GNOME Settings บนเครื่องจริงโชว์ "Never" ขอให้ตรวจสอบ — สืบสวนร่วมกับผู้ใช้ผ่าน SSH หลายรอบ (`loginctl`, `dconf dump`, รัน Python function เดียวกับที่ VAS ใช้ตรงๆ ผ่าน `PYTHONPATH`, วน loop อ่านค่าซ้ำ 20 รอบ) จนพิสูจน์ได้ชัดว่า **ค่าจริงที่ persist อยู่คือ `0` (Never) มาตั้งแต่ 14:07 ไม่เคยเปลี่ยนเลย** (ยืนยันด้วย `dconf dump`, อ่านซ้ำ 20 รอบต่อเนื่องได้ `uint32 0` ทุกรอบ, `~/.config/dconf/user` mtime ไม่ขยับ) — ค่า `180` (ที่ UI เคยโชว์) และ `32` (ที่อ่านได้ครั้งเดียวตอนทดสอบ) เป็นการอ่านผิดพลาดชั่วคราว เกิดใกล้ๆ ช่วงที่ session/VAS process เพิ่งเริ่มใหม่ๆ ทั้งคู่ (race window ตอน dconf-service/D-Bus ยังไม่ warm up เต็มที่)
- root cause อยู่ที่ `parse_gsettings_uint()`: ใช้ `re.search(r"(\d+)", output)` หาตัวเลขตัวแรกที่เจอ**ที่ไหนก็ได้**ในสตริง `stdout` ทั้งก้อน — ถ้าช่วง race นั้น `gsettings get` คืน output ที่ไม่ใช่รูปแบบปกติ `uint32 N` เลย (returncode ยังเป็น 0 ด้วยซ้ำ ไม่ error ให้เห็น) regex จะหยิบเลขอะไรก็ได้ที่ปนอยู่มาใช้เป็นค่าจริงไปเลย ไม่มีการ validate รูปแบบ
- `src/features/display/display.py`: เปลี่ยน `parse_gsettings_uint()` จาก `re.search()` เป็น `re.fullmatch(r"(?:uint32\s+)?(\d+)", output.strip())` — บังคับให้ทั้งสตริง (หลัง strip whitespace) ต้องตรงรูปแบบ `uint32 N` หรือ `N` ล้วนๆ เท่านั้น ถ้าไม่ตรงเป๊ะให้ถือว่าอ่านไม่สำเร็จ คืน `None` (หน้าเว็บจะโชว์ "ไม่ทราบค่า" ผ่าน `screen_blank_label(None)` อย่างตรงไปตรงมา แทนที่จะโชว์ตัวเลขมั่วๆ ที่ดูน่าเชื่อถือแต่ผิด) — ไม่ได้แก้จุดอื่นเลย เพราะ `get_gnome_screen_blank_seconds()`/`DisplayCommandRunner`/session detection ทั้งหมดตรวจสอบแล้วว่าทำงานถูกต้องอยู่แล้ว ไม่ใช่สาเหตุ
- **ตรวจสอบ**: เขียน unit test แบบ standalone ใน `outputs/parse_check.py` (นอกโปรเจกต์ ไม่ commit) ทดสอบ `"uint32 0"`, `"uint32 300"`, `"300"`, `"  uint32 32  "` (ต้องผ่าน) กับ `""`, `"(process:1234): GLib-GIO-CRITICAL **: ... 21:38:32"` (จำลอง output แบบ race ที่มี timestamp ปน), `"Error: no such key"` (ต้องคืน `None`) — ผ่านครบทุกเคส จำลองสถานการณ์ที่เจอจริงได้ตรง (เคสที่มี timestamp ปนคือ pattern เดียวกับที่น่าจะทำให้ได้ `32` มาก่อนหน้านี้) — เจอปัญหา mount-sync ค้างที่ `display.py` อีกครั้ง (เจอทุกรอบที่แก้ไฟล์นี้ตลอดทั้ง session) ยืนยันผ่าน Read tool ว่าไฟล์จริงถูกต้องครบ, `py_compile` ผ่าน bash ให้ syntax error ปลอมเหมือนทุกครั้ง — **ยังไม่ได้ทดสอบบนเครื่องจริงว่า UI โชว์ "ไม่ทราบค่า" ถูกต้องตอนเจอ race จริง** เพราะ race นี้เกิดไม่บ่อยและ reproduce ยาก (ต้องจับจังหวะตอน session เพิ่งเริ่ม) — แนะนำให้สังเกตต่อไปว่าหลัง deploy แล้วยังเจอค่าผิดปกติแบบนี้อีกไหม ถ้าเจอ "ไม่ทราบค่า" บ่อยเกินคาด (ไม่ใช่แค่ตอน session เพิ่ง login) อาจต้องพิจารณาเพิ่ม retry เข้าไปใน `get_gnome_screen_blank_seconds()` แทนที่จะแค่ fail แล้วคืน `None` ครั้งเดียว

### monitors.xml sync-on-apply (ระดับเครื่อง ↔ ระดับ user) + ลบ helper text screen blank + โชว์ config file ระดับ user
- **ที่มา**: ต่อจาก entry ด้านล่าง (สถานะ login/blank screen) — ผู้ใช้ส่งภาพหน้าจอ `/display` จริง 3 รูป (Monitor tab, Config Files tab, การ์ด monitors.xml) พร้อม 3 คำขอ: (1) เอา helper text ใต้ dropdown Screen Blank ออก เพราะซ้ำซ้อนกับแถบสถานะบนสุดที่เพิ่งเพิ่มไป (2) เช็คว่ามีไฟล์ config สำหรับ screen blank ไหม ถ้ามีให้โชว์ใน Config Files (3) คำถามหลัก — ผู้ใช้เป็น support ที่ตั้งค่าจอผ่านเว็บ VAS จากระยะไกล แต่ทีมหน้างานอาจไปปรับผ่าน GNOME Settings บนตัวเครื่องทีหลัง ทำให้ `~/.config/monitors.xml` ของ kiosk user (ชนะ `/etc/xdg/monitors.xml` เสมอตาม mutter policy) ทับค่าที่ VAS ตั้งไว้แบบเงียบๆ — สั่งวิเคราะห์+วางแผนก่อน (ห้าม implement) แล้วเลือกแนวทาง **"A. Sync-on-apply"** จาก 3 ทางเลือกที่เสนอ (A: sync ทุกครั้งที่ Apply, B: full lock ด้วย chattr +i, C: toggle เลือกต่อ user) ผ่าน AskUserQuestion จากนั้นสั่ง implement
- `src/web/templates/display.html`: ลบ helper text 3 branch (gnome/openbox/else) ใต้ `screenBlankSelect` เหลือแค่ dropdown เดี่ยวๆ — screen blank ไม่มี config file แยกให้โชว์ (xset persist ใช้ไฟล์เดียวกับ `.xprofile` อยู่แล้ว, GNOME เขียนตรงเข้า live dconf database ไม่ใช่ text file)
- `src/features/display/monitors_xml.py`: เพิ่ม `MonitorsXmlManager.write_user_level(state, rotation, home, username)` เขียน `~/.config/monitors.xml` ของ kiosk user ด้วยเนื้อหาเดียวกับที่เขียนระดับเครื่อง (`build_monitors_xml()` ตัวเดิม) แล้ว chown ให้ user เป้าหมาย — เพิ่ม `remove_user_level(home)` ลบไฟล์นี้ตอนปิด toggle Persist **เฉพาะไฟล์ที่มี `MONITORS_XML_SIGNATURE`** (คือไฟล์ที่ VAS เขียนเอง) กันลบไฟล์จริงของ user ที่หน้างานตั้งเองผ่าน GNOME Settings โดยไม่ตั้งใจ — เพิ่ม private helper `_chown_path_to_user()` (สำเนาเจตนาจาก `display.py` ตัวเดิม ไม่ import ข้ามไฟล์ผูก private helper กัน)
- `src/server.py`: `_write_system_monitors_xml()` เปลี่ยน return type จาก `str | None` เป็น `tuple[str | None, str | None]` (`error` fatal สำหรับ caller คืน 500, `sync_warning` non-fatal สำหรับ caller โชว์เตือนแต่ไม่ fail request) — หลังเขียนระดับเครื่องสำเร็จ resolve `kiosk_target_user` ผ่าน `resolve_kiosk_target_user(list_kiosk_linux_users())` (คนละ resolve mechanism จาก `query_user` ที่ใช้ query D-Bus — คนละคนกันได้ แต่ state เป็นค่าฮาร์ดแวร์ทั่วไปเขียนไปที่ home ใครก็ได้) แล้วเรียก `write_user_level()` sync ตาม — ทั้ง 2 caller (`/api/display/apply`, `/api/display/persist-write`) unpack tuple ใหม่ ส่ง `monitorsXmlUserSync` เพิ่มในทุก response — `/api/display/persist-remove` (target `monitors_xml`) เรียก `remove_user_level()` คู่กับ `remove_system_level()` เดิม กัน user-level ที่ sync ไว้ค้างเป็นตัวชนะต่อไปทั้งที่ปิด Persist แล้ว — `_allowed_config_paths()` เพิ่ม key `monitors_xml_user` แบบ dynamic (ขึ้นกับ kiosk_target ปัจจุบัน ใส่เฉพาะ resolve ได้จริง ไม่ fallback เป็น path ปลอม) — `GET /display` เพิ่ม `monitors_xml_user=collect_monitors_xml_system_status(user_monitors_xml_path(kiosk_target.home))` ส่งเข้า template (ฟังก์ชันเดิม generic บน path อยู่แล้ว)
- `src/web/templates/display.html`: การ์ดเตือนสีเหลือง "มี monitors.xml ของตัวเองอยู่แล้ว — ค่านี้จะไม่มีผล" เปลี่ยนเป็นสีน้ำเงิน/informational อธิบายว่า VAS จะ sync ให้อัตโนมัติทุกครั้งที่ Apply/Persist แต่ถ้าหน้างานปรับผ่าน GNOME Settings ทีหลัง ค่านั้นจะชนะจนกว่าจะกด Apply/Persist จาก VAS รอบใหม่ — เพิ่ม card ใหม่ `monitors_xml_user` ใน Config Files tab (โชว์เฉพาะตอน `kiosk_target_has_own_monitors_xml`) แสดง `has_signature` ของไฟล์นี้บอกได้ว่าไฟล์ล่าสุด "VAS sync ไว้" หรือ "หน้างานตั้งเองไม่ผ่าน VAS" — `setResult()` เพิ่ม status `"warning"` (สี `text-caution`) — `doApply()`/`bindPersistToggle()` โชว์ `monitorsXmlUserSync` เป็น warning แยกจาก error หลัก (ไม่ fail ทั้ง request เพราะระดับเครื่องเขียนสำเร็จไปแล้ว)
- **ตรวจสอบ**: เจอปัญหา mount-sync ค้างรุนแรงกว่าทุกรอบก่อนหน้าในเซสชันนี้ — คราวนี้ค้างทั้ง `monitors_xml.py`, `server.py` (ค้างที่ 3986 บรรทัด ทั้งที่ไฟล์จริงยาว 4165 บรรทัด — syntax error ที่ bash รายงานพิสูจน์แล้วว่าไม่จริงหลัง Read tool ยืนยันเนื้อหาบริเวณนั้นถูกต้องสมบูรณ์) **และครั้งแรกที่ค้างแม้แต่ไฟล์ scratch ใน `outputs/` ที่เขียน/แก้เองผ่าน Write/Edit tool โดยตรง** (`monitors_xml_check.py` ค้างที่ 530 บรรทัดพร้อม byte ภาษาไทยเพี้ยนกลางบรรทัด ทั้งที่ Read tool ยืนยันไฟล์จริงยาว 546 บรรทัดและถูกต้องครบ) — สรุปว่าเป็นบั๊ก mount-sync ระดับ session ไม่ใช่เฉพาะไฟล์ในโปรเจกต์ — verification หลักจึงทำผ่าน Read tool อ่านทุก region ที่แก้ไขแบบเต็ม (ไม่ใช่แค่ py_compile/ruff ผ่าน bash) ยืนยัน bracket/quote/indentation ถูกต้องครบทุกจุด, `monitors_xml.py` ฉบับ reconstruct ผ่าน `py_compile`+`ruff check` สำเร็จ (ก่อนเพิ่ม `remove_user_level` — ส่วนที่เพิ่มทีหลังตรวจด้วย Read tool อย่างเดียวเพราะ bash ค้างซ้ำอีกรอบ), `server.py` ตรวจด้วยการแยก edit regions ออกมาเป็นไฟล์เดี่ยวแล้ว `py_compile` ผ่าน (ครอบคลุม logic ทั้งหมดที่แก้แต่ไม่ใช่ full-file compile เพราะไฟล์ใหญ่เกินจะ reconstruct ทั้งไฟล์ในบทสนทนานี้อย่างคุ้มค่า), `display.html` อ่านผ่าน Read tool ครบทั้งไฟล์ (1242 บรรทัด) ยืนยันปิด `{% endblock %}` ทั้ง 2 จุดถูกต้อง — **ยังไม่ได้ทดสอบบนเครื่องจริง**: ต้องเปิดหน้า `/display` จริง กด Apply ตอน kiosk user มี `~/.config/monitors.xml` ของตัวเองอยู่ก่อน แล้วตรวจว่าไฟล์นั้นถูกเขียนทับด้วยค่าที่ตั้งจาก VAS จริง (มี `MONITORS_XML_SIGNATURE`), ตรวจว่า `remove_user_level()` ลบเฉพาะไฟล์ที่ VAS เขียนเองจริงไม่ลบไฟล์ของหน้างาน, ตรวจว่า card `monitors_xml_user` ใน Config Files โหลดเนื้อหาได้ถูกต้องผ่าน `/api/display/config-content?key=monitors_xml_user`

### เพิ่มการตั้งค่า Screen Blank สำหรับ session แบบ GNOME ผ่าน gsettings (`xset` เดิมไม่มีผลกับ GNOME เลย)

### แก้ป๊อปอัพแปลภาษายังขึ้นทั้งที่ปิด flag แล้ว + เพิ่มปิดปัดซ้าย/ขวา + ล็อกทางหลุดออกจาก kiosk ของ GNOME Shell
- **ที่มา**: ผู้ใช้รายงานว่าเปิด toggle "ปิดป๊อปอัพเสนอแปลภาษา" ในหน้าคีออสแล้ว แต่ป๊อปอัพแปลภาษายังขึ้นอยู่บนเครื่องจริง (Ubuntu 22.04, VirtualBox) พร้อมขอให้ตรวจสอบ kiosk mode ทั้งหมดว่ามีอะไรขาด และขอเพิ่มปิดการปัดซ้าย/ขวาย้อนกลับ-ไปหน้าถัดไปทั้งในเบราว์เซอร์และระดับ Ubuntu OS — วางแผนก่อน (ไม่ implement) แล้วผู้ใช้สั่งให้ทำตามแผนจริง โดยเจาะจงให้ focus เฉพาะ GNOME (ข้าม Openbox ซึ่งถูกพักฟีเจอร์ไว้อยู่แล้วจาก entry ก่อนหน้า)
- root cause ของป๊อปอัพแปลภาษา: `--disable-translate` เป็น command-line switch ที่ Chromium เลิกรองรับไปแล้วในหลายเวอร์ชัน (อ้างอิง Chromium issue 41347677/770776 — "command line disable-translate option is ignored") สคริปต์ที่ VAS สร้างยังใส่ flag นี้อยู่ แต่ Chromium มองข้ามเงียบๆ ไม่ error ไม่เตือน — ไม่รู้เวอร์ชัน Chromium จริงบนเครื่อง (snap chromium/apt chromium-browser) จึงใส่ทั้ง switch เดิมและ `--disable-features=Translate` (วิธีที่ยืนยันว่าใช้ได้ในเวอร์ชันปัจจุบัน) พร้อมกันเผื่อไว้
- `src/features/kiosk/manager.py::CHROME_KIOSK_FLAG_DEFS`: เพิ่ม key `"features"` ในทุก item (ค่า default `()`) แยกจาก `"flag"` เดิม — `disable_translate` เพิ่ม `features=("Translate",)`, เพิ่ม item ใหม่ 2 ตัว: `disable_swipe_navigation` (`--overscroll-history-navigation=0` + `--disable-features=OverscrollHistoryNavigation,TouchpadOverscrollHistoryNavigation` — ปิดปัดซ้าย/ขวาในเบราว์เซอร์ ชื่อ feature เปลี่ยนไปมาในแต่ละเวอร์ชัน Chromium จึงใส่ทั้งคู่) และ `disable_pinch_zoom` (`--disable-pinch` กัน pinch-to-zoom บนจอสัมผัส)
- `build_kiosk_launch_script()`: รวม `features` จากทุก toggle ที่เปิดอยู่เป็น `--disable-features=A,B,C` **ตัวเดียว** ต่อท้ายคำสั่ง แทนที่จะให้แต่ละ toggle ยิง `--disable-features` ของตัวเองแยกกัน — Chromium ใช้แค่ occurrence สุดท้ายของ switch ซ้ำบน command line เดียวกัน (last-one-wins) ถ้าไม่ merge จะมี toggle หนึ่งถูกเมินไปเงียบๆ ทันทีที่เปิดพร้อมกันสองตัวขึ้นไป
- เพิ่ม GNOME lockdown (gsettings) — เฉพาะ `session_type == "gnome"` เท่านั้นตามที่ผู้ใช้ขอ: `GNOME_LOCKDOWN_FLAG_DEFS` (3 toggle: ปิด Hot Corner, ปิดคีย์ลัดเปิด Terminal Ctrl+Alt+T, ปิดปุ่ม Super เปิด Activities Overview — ทั้งหมด default เปิด), `build_gnome_lockdown_preamble()` สร้าง preamble รัน `gsettings set ...` แทรกเข้า `kiosk-launch.sh` ผ่าน `_insert_after_shebang()` แบบเดียวกับ Openbox preamble ที่มีอยู่แล้ว — รันคำสั่งในบริบท session ของ kiosk user เองตอน login (มี D-Bus/X session ให้ gsettings เขียนค่าได้จริง) แทนที่จะเรียกจาก VAS server (root, ไม่มี session กราฟิกผูกอยู่ เรียก gsettings ตรงๆ จะ fail เงียบๆ), `_parse_gnome_lockdown_flags()` อ่านสถานะกลับจากไฟล์บนดิสก์แบบเดียวกับ `_parse_kiosk_flags()` — เพิ่ม field `gnome_lockdown_flags` ใน `KioskAutostartStatus` (ค่าเฉพาะ session แบบ gnome เท่านั้น Openbox คงค่า default ไว้เฉยๆ ไม่ถูกอ่าน/เขียนจริง) และพารามิเตอร์ใหม่ใน `write_autostart()`/`collect_kiosk_autostart_status()`

### เพิ่มการตั้งค่า Resolution + Refresh Rate ในหน้า `/display` ผ่าน GNOME/mutter (`ApplyMonitorsConfig`) พร้อมเขียนลง `monitors.xml`
- **ที่มา**: ผู้ใช้ขอให้เพิ่มการตั้งค่า Resolution + Refresh Rate ลงในโปรแกรม VAS โดยให้ mirror ตาม GNOME Settings > Displays จริง (ส่งภาพหน้าจอมาให้ดูตัวอย่าง Dropdown "1920 × 1080 (16:9)" / "60.00 Hz") และย้ำว่าต้องตั้งค่าผ่าน GNOME + mutter (ไม่ใช่ยิง `xrandr --mode` ตรงๆ) พร้อมเขียนผลลัพธ์ลง `monitors.xml` ด้วย — เหตุผลเดียวกับที่ rotation ต้องเขียนผ่าน D-Bus แทน xrandr ตรงๆ ในฟีเจอร์ก่อนหน้า (session แบบ GNOME จะเขียนทับค่าที่ตั้งผ่าน xrandr ตรงๆ กลับเป็นเดิมอยู่ดี)

- `src/server.py`: เพิ่ม route `GET /api/display/monitors-xml/modes` คืนรายการ mode ทั้งหมดที่จอรองรับจริง (query อย่างเดียวไม่ apply) ให้หน้าเว็บเอาไปเติม dropdown, เพิ่ม `_apply_monitor_resolution()` helper (requery serial สดของตัวเองเสมอ ไม่รับ serial จาก frontend, เช็ค session เป็น Openbox ก่อนเสมอแล้วคืน error เฉพาะเจาะจงถ้าใช่ — เหตุผลเดียวกับ `_write_system_monitors_xml()`), เพิ่ม `resolutionModeId` เข้า `/api/display/apply` — ถ้ามีค่าจะเรียก `_apply_monitor_resolution()` ก่อน `configurator.apply_runtime()` เสมอ แล้วให้ `_write_system_monitors_xml()` เดิม (ถ้า `persistMonitorsXml` เปิดอยู่) query `GetCurrentState` ซ้ำเอาความละเอียดใหม่ไปเขียนต่อเองโดยไม่ต้องแก้ฟังก์ชันนั้นเลย

- **ตรวจสอบ**: เจอปัญหา mount-sync ค้างรวม 3 ครั้งระหว่างแก้รอบนี้ — `base.html` เจอ trailing null bytes ท้ายไฟล์แบบเดิม (ตัดออกด้วยการ truncate ตรงจุดที่ null เริ่ม), `kiosk.html` เจอการตัดขาดกลางไฟล์แบบรุนแรงกว่าที่เคยเจอ (ไฟล์ในแซนด์บ็อกซ์ขาดหายไปทั้งท่อนตั้งแต่กลางฟังก์ชัน `saveAutostart()` ถึงท้ายไฟล์ ~220 บรรทัด แม้ยังเป็น UTF-8 ที่ decode ผ่านได้ปกติเพราะขาดพอดีตรงขอบบรรทัด ไม่ใช่กลาง multi-byte character แบบที่เจอมาก่อน — ตรวจพบเพราะ `jinja2.Environment().parse()` แจ้ง "Unexpected end of template" ทั้งที่ decode/wc -l ดูปกติ), `display.html` ก็เจอการตัดขาดกลางไฟล์แบบเดียวกันหลังแก้ toast wiring (ขาดตั้งแต่กลางฟังก์ชัน `simStart()`) — ทั้งสองไฟล์แก้ด้วยเทคนิค byte-splice เดิม: หา anchor string ที่ไม่ซ้ำในไฟล์ (ก่อนจุดที่ขาด), ตัด prefix ที่ยืนยันถูกต้องแล้วไว้, ต่อ tail ที่พิมพ์ใหม่จาก Read tool (ยืนยันเนื้อหาถูกต้อง) เข้าไปแทนให้ตรงกับ line-ending เดิมของไฟล์ (`kiosk.html`/`base.html` ใช้ `\r\n`, `display.html` ใช้ `\n`) — หลัง reconstruct ทั้งสามไฟล์: `jinja2.Environment().parse()` ผ่านครบทั้ง `kiosk.html`, `display.html`, `base.html`, ไม่มี null byte เหลือ, `python3 -m py_compile` ผ่านสำหรับไฟล์ Python ที่เกี่ยวข้อง (ไม่มีการแก้ `.py` ในรอบนี้ เป็นการแก้ template+JS ล้วนๆ) — **ยังไม่ได้เปิดหน้าเว็บจริงทดสอบ**: ต้องเปิดหน้า `/display` ทดสอบกดปุ่ม Save/Apply ต่างๆ ดูว่า toast ขึ้นสีทึบถูกต้อง และเปิดหน้าคีออสทดสอบว่าเลือก Openbox ไม่ได้จริง, ปุ่ม "+ เพิ่ม" เปิด modal ถูกต้อง, และปิด modal ด้วย Escape/backdrop/ปุ่มยกเลิกทำงานครบ

### แก้ toggle "Persist touch ใน Xorg" ไม่เคยมีผลจริงหลัง reboot — ตั้งค่า X input property ผิดตัวมาตั้งแต่แรก
- **ที่มา**: ผู้ใช้ทดสอบหมุนจอ (rotate left) บน VM (`ubuntu2204-first-test`, VirtualBox) แล้วรายงานว่า touch ยังเพี้ยนอยู่แม้ตั้งค่าผ่านหน้าเว็บครบแล้ว — ไล่ debug ทีละชั้นร่วมกับผู้ใช้ผ่าน CLI จริงบนเครื่อง: (1) ยืนยัน `/etc/xdg/monitors.xml` ถูกต้อง (`rotation=left`, เขียนโดย D-Bus fix ของ entry ก่อนหน้า) (2) ยืนยัน X server หมุนจอถูกต้องสมบูรณ์แบบจริง (`xrandr --query` โชว์ `current 1080 x 1920`, `Virtual1 ... left`) ตัดความเป็นไปได้เรื่องจอออกหมด (3) เช็ค touch device พบว่า `Coordinate Transformation Matrix` (property ที่ `apply_runtime()`/`.xprofile` ใช้จริงตอน set live) ยังเป็น identity `1 0 0 0 1 0 0 0 1` แม้ persist ไฟล์ `99-vending-touchscreen.conf` จะมีค่าที่ถูกต้อง (`0 -1 1 1 0 0 0 0 1`) อยู่แล้วก็ตาม แม้ reboot เครื่องจริงแล้วก็ยังไม่มีผล (4) เช็ค property `libinput Calibration Matrix` (คนละตัวกับข้อ 3) พบว่ามีค่าที่ถูกต้องตรงกับไฟล์ conf เป๊ะ — สรุปได้ว่าไฟล์ conf ถูกอ่านจริงตอน X server เริ่มทำงาน แต่ไปตั้งค่า property ผิดตัว
- root cause: `build_xorg_touchscreen_config()` (`src/features/display/display.py`) ใช้ `Option "CalibrationMatrix"` ซึ่งเป็น option เฉพาะของ driver libinput ที่ map ไปตั้งค่า property `"libinput Calibration Matrix"` (ใช้ calibrate touch แบบ raw-sensor-to-normalized ไม่เกี่ยวกับทิศทางจอ) — ไม่ใช่ `"Coordinate Transformation Matrix"` ที่ `apply_runtime()` และ `display-session.sh` (`.xprofile`) ใช้จริงตอน set ค่า live ผ่าน `xinput set-prop` ทำให้ toggle "Persist touch ใน Xorg" โชว์สถานะ "ตั้งค่าแล้ว" ในหน้าเว็บตลอด แต่ไม่เคยมีผลจริงกับ touch หลัง reboot เลยตั้งแต่ฟีเจอร์นี้ถูกสร้างขึ้นมา (ไม่เกี่ยวกับงาน monitors.xml ของ entry ก่อนหน้าในวันก่อน)
- แก้เป็น `Option "TransformationMatrix"` ซึ่งเป็น option มาตรฐานที่ X server map ไปตั้งค่า `"Coordinate Transformation Matrix"` ตรงกับกลไกอื่นๆ ในโค้ดเดียวกัน — อัปเดต `tests/test_display.py::test_xorg_config_includes_signature_and_touch_name` และตัวอย่างใน `docs/display-commands.md`/`docs/kiosk-user-monitor-rotation-investigation.md` ให้ตรงกัน (เพิ่ม note แก้ไขภายหลังในเอกสารสืบสวนเดิมแทนการลบของเก่าทิ้ง)
- **ตรวจสอบ**: ระหว่างแก้เจอ mount-sync ค้างอีก 2 รอบ (`display.py` ค้าง 784 บรรทัดแทนที่จะเป็น 837, และพบว่า `tests/test_display.py` มีบั๊ก truncation ที่มีมาก่อนหน้านี้แล้วจริงในไฟล์ต้นฉบับเอง ไม่ใช่แค่ mount-sync — บรรทัดสุดท้าย `assert "managed\n" not in conte` ขาดหายเป็น `NameError` ซึ่งน่าจะเป็นร่องรอยของบั๊ก mount-sync เดียวกันนี้จาก session ก่อนหน้าที่ไม่เคยถูกจับได้ เพราะไม่เคยมีใครรัน `pytest` จริงมาก่อน) — reconstruct `display.py` ผ่าน `outputs/` แล้ว `cp` ทับ, แก้ truncation ใน `test_display.py` ให้สมบูรณ์ (`conte` → `content`) — `python3 -m py_compile` ผ่านทั้งคู่ — รัน `pytest tests/test_display.py` จริงเป็นครั้งแรกในรอบนี้ (ต้องใช้ `PYTHONPATH=src:src/features/display` เพราะไฟล์ test ใช้ bare import `from display import` ซึ่งไม่ตรงกับ convention `PYTHONPATH=src` ที่ระบุใน README เฉยๆ — ปัญหานี้มีมาก่อนแล้ว ไม่ได้แก้ในรอบนี้เพราะอยู่นอกขอบเขตงาน) ผลคือ **ผ่านทั้ง 11 เทสต์** รวมถึงเทสต์ที่ตรวจ `TransformationMatrix` โดยตรง — `mypy --strict --cache-dir=/tmp/mypy_cache`: `display.py` มี error 2 รายการเท่าเดิม (unused type:ignore ใน `_chown_to_effective_user()` ที่ไม่เกี่ยวกับจุดที่แก้ ยืนยันจาก entry ก่อนหน้าว่าเป็น pre-existing) รวมกับ baseline `server.py` 60 error เป็น 62 ไม่มี error ใหม่จากจุดที่แก้เลย
- **ตรวจสอบเพิ่มเติม (end-to-end บนเครื่องจริง หลัง deploy + reboot)**: ผู้ใช้ deploy โค้ดใหม่, เปิด toggle ใหม่ (ไฟล์ `99-vending-touchscreen.conf` ยืนยันผ่านหน้าเว็บว่ามี `Option "TransformationMatrix" "0 -1 1 1 0 0 0 0 1"` ถูกต้อง), reboot แล้วเช็ค `xinput list-props` จริง: `Coordinate Transformation Matrix` = `0 -1 1 1 0 0 0 0 1` ตรงกับค่าที่ต้องการเป๊ะ (`libinput Calibration Matrix` กลับไปเป็น identity ตามที่ควรจะเป็น เพราะเลิกยุ่งกับ property นี้แล้ว) — **ยืนยันว่า root cause ที่แท้จริง (property name ผิด) แก้ถูกจุดแล้วจริง ปิดเคสฝั่งโค้ดได้**
- อย่างไรก็ตาม ผู้ใช้รายงานว่าแตะหน้าจอจริงยังรู้สึกเพี้ยนอยู่แม้ matrix จะถูกต้องแล้ว — วิเคราะห์ร่วมกันแล้วสรุปว่าน่าจะไม่ใช่บั๊กโค้ดอีกต่อไป แต่เป็นข้อจำกัดเฉพาะของการทดสอบผ่าน **VirtualBox USB Multi-Touch** (virtual absolute-pointer device ของ VirtualBox Guest Additions) ซึ่งน่าจะรายงานพิกัดสัมผัสโดยอ้างอิงความละเอียดจอที่ VirtualBox เข้าใจเอง (ผ่าน host-to-guest coordinate mapping ของตัว hypervisor) ไม่ใช่ raw coordinate จาก physical digitizer โดยตรงแบบทัชสกรีนจริง — ถ้า VirtualBox ยังไม่รู้ว่าจอ guest หมุนไปแล้ว (ตามที่พบไปแล้วในปัญหาจอบีบ/ครอปของ gnome-initial-setup ก่อนหน้านี้ในวันเดียวกัน) พิกัดที่ virtual touch device รายงานมาก็จะอิงกรอบอ้างอิงเดิมที่ยังไม่หมุน ทำให้เกิดการแมพซ้อนสองชั้นที่ผิดเพี้ยน แม้ `Coordinate Transformation Matrix` ฝั่ง X11 จะถูกต้องแล้วก็ตาม — เคสนี้ไม่ควรเกิดกับทัชสกรีนจริงที่ต่อกับเครื่อง vending เพราะ digitizer จริงส่ง raw coordinate ตรงเข้า X11 ไม่ผ่านชั้น translation ของ hypervisor แบบนี้ — **ยังไม่ได้ทดสอบยืนยันบนเครื่อง vending จริง (จอทัชสกรีนจริง + touch panel จริง)** ต้องทดสอบบนฮาร์ดแวร์จริงเพื่อยืนยันว่า fix นี้แก้ปัญหาได้สมบูรณ์ ก่อนถือว่าปิดเคส "touch เพี้ยน" ทั้งหมด

- **ที่มา**: สืบสวนร่วมกับผู้ใช้จากเคสทดสอบบน VM (`ubuntu2204-first-test`, Ubuntu 22.04 + GDM3, X11 session) — หน้าเว็บ VAS ขึ้น "Monitors: none detected" ทั้งที่ `xrandr --query` รันเองในฐานะ user ที่ login อยู่ (`first`) เจอ `Virtual1 connected` ปกติดี พิสูจน์ root cause ด้วยการรัน `loginctl show-session <id> -p Display` (คืนค่าว่าง) และ `ls -la ~/.Xauthority` ของ user นั้น (ไม่มีไฟล์อยู่จริง) — Ubuntu 22.04 ที่ session ผ่าน GDM3 จะไม่สร้าง `~/.Xauthority` แบบดั้งเดิม (แบบ `startx`/`xinit`) แต่เก็บ auth cookie ต่อ session ไว้ที่ `/run/user/<uid>/gdm/Xauthority` แทน — `_xauthority_for_user()` เดิมเช็คแค่ `home/.Xauthority` อย่างเดียว ได้ `None` กลับมาเสมอสำหรับ session แบบนี้ → `_default_xauthority()` fallback ไปเช็ค `Path.home()` ของ **root** (process ของ VAS server เอง ซึ่งรันเป็น systemd service ไม่มี `User=` กำหนด) ซึ่งไม่มี X session ของตัวเองอยู่แล้ว → XAUTHORITY เป็น `None` ทั้งหมด → `xrandr --query` ที่ VAS เรียกเชื่อม X server แบบไม่มี auth cookie → error "Can't open display" → stdout ว่างเปล่า → `parse_xrandr_outputs()` ไม่เจอบรรทัด `connected` เลย → หน้าเว็บเข้าใจผิดว่าไม่มีจอต่ออยู่
- แก้ `_xauthority_for_user()` ให้เช็คหลาย candidate path ตามลำดับความสำคัญแทนเช็คจุดเดียว: `/run/user/<uid>/gdm/Xauthority` (GDM3 session — ลำดับแรก) ก่อน แล้วค่อย fallback ไป `home/.Xauthority` (session แบบ `startx`/`xinit` หรือ display manager อื่นที่ยังสร้างไฟล์นี้ในโฮมไดเรกทอรี) — ใช้ `pwd.getpwnam()` ดึงทั้ง `pw_dir` และ `pw_uid` พร้อมกันจากครั้งเดียว (เดิมดึงแค่ `pw_dir`)
- **ตรวจสอบ**: `python3 -m py_compile` ผ่าน (บนสำเนาที่ตัด null-byte padding ท้ายไฟล์ที่ bash mount เติมมาเองออกก่อน — เจอปัญหา mount-sync เดิมของ session นี้ซ้ำอีกครั้ง ดู entry `[2026-07-05]`/`[2026-07-09]` เรื่องเดียวกัน คราวนี้ปรากฏเป็น trailing null bytes 30 ตัวแทนที่จะเป็นการตัดท้ายไฟล์ขาดหาย — ยืนยันเนื้อหาไฟล์จริงถูกต้องครบผ่าน Read tool เทียบกับสำเนาที่ทำความสะอาดแล้ว ตรงกัน 100%) — `mypy --strict` (ติดตั้ง `flask` ในแซนด์บ็อกซ์ก่อนเพื่อลด noise จาก decorator ที่ยังไม่มี stub) รันบนสำเนาที่สะอาดแล้ว: ฟังก์ชันที่แก้ (`_xauthority_for_user`) ไม่มี error ใหม่เลย จำนวน error รวมทั้งไฟล์ลดลง 1 รายการ (ลบ `# type: ignore[attr-defined]` ที่ไม่จำเป็นแล้วออกจากบรรทัดเดิม) — **ทดสอบ end-to-end บนเครื่องจริงแล้วพบว่ายังไม่พอ** ดู entry ถัดไปวันเดียวกัน

### `_default_x_display()` (`src/server.py`) fallback ผิดเป็น `:0` แบบ hardcode — DISPLAY จริงบนเครื่องทดสอบคือ `:1`
- **ที่มา**: หลัง deploy fix ของ `_xauthority_for_user()` ข้างบนแล้ว หน้า "จอแสดงผล" ยังขึ้น "none detected" เหมือนเดิมแม้ restart service สำเร็จ (`vending-auto-setup-server.service`, PID เปลี่ยนแล้ว) — proof ทีละชั้น: (1) เช็คโค้ดที่ deploy จริงมี fix อยู่ ✅ (2) เช็ค `/run/user/1000/gdm/Xauthority` มีไฟล์จริง ✅ (3) รัน `echo $DISPLAY` ในเซสชัน GUI ของ user `first` จริง ได้ `:1` ไม่ใช่ `:0` — จำลองคำสั่งแบบเดียวกับ VAS ในฐานะ root: `DISPLAY=:0 ...` ล้มเหลว แต่ `DISPLAY=:1 XAUTHORITY=/run/user/1000/gdm/Xauthority xrandr --query` สำเร็จเห็น `Virtual1 connected` ทันที — ยืนยัน root cause คือ `_default_x_display()` เดิม fallback เป็น `":0"` คงที่เมื่อ `loginctl show-session -p Display` คืนค่าว่าง (ซึ่งเกิดขึ้นจริงบนเครื่องนี้) ทั้งที่ X server จริงรันอยู่ที่ `:1`
- เพิ่ม `_find_display_for_user()` (`src/server.py`) — หา DISPLAY จริงจาก `/proc/<pid>/environ` ของ process ใดๆ ที่ user นั้นเป็นเจ้าของ (root อ่าน `/proc/*/environ` ของ user อื่นได้เสมอไม่ติด permission) แทนการเดาค่าคงที่ — `_default_x_display()` เรียกฟังก์ชันนี้เป็นลำดับที่ 2 (หลังเช็ค `loginctl` Display property ปกติ) ก่อนจะ fallback ไป `:0` เป็นทางเลือกสุดท้ายจริงๆ
- **ตรวจสอบ**: เจอปัญหา mount-sync ซ้ำอีกครั้ง (ครั้งนี้ตัดท้ายไฟล์ขาดหายกลางบรรทัด `for candidate in candidates:` พอดี ไม่ใช่ null-byte padding เหมือนรอบก่อน) — reconstruct ตาม convention ที่โปรเจกต์เคยบันทึกไว้ (entry `[2026-07-05]`): เทียบ prefix ที่ยังสมบูรณ์ (มั่นใจได้ถึงบรรทัดที่ตัดขาดพอดี) กับเนื้อหาที่ถูกต้องจาก Read tool แล้วเขียนทับกลับผ่าน bash โดยตรง (ไม่ผ่าน Edit tool ซ้ำ) — `python3 -m py_compile` ผ่านหลัง reconstruct, `mypy --strict` (บนสำเนาที่ตัด null-byte padding รอบใหม่ที่โผล่มาอีกหลัง edit ครั้งถัดไปออกแล้ว): ฟังก์ชันที่แก้ทั้งสอง (`_default_x_display`, `_find_display_for_user`) ไม่มี error ใหม่เลย จำนวน error รวมทั้งไฟล์คงที่ 60 รายการเท่าเดิม — **ทดสอบ end-to-end บนเครื่องจริงแล้วพบว่ายังไม่พอ** ดู entry ถัดไปวันเดียวกัน

### `_default_x_display()` ยัง fallback ผิดอยู่ดี — `loginctl show-session -p Display` เชื่อถือไม่ได้ (คืนค่าได้ทั้งว่างเปล่าและ ":0" ผิดๆ คนละครั้งกัน)
- **ที่มา**: deploy fix ของ `_find_display_for_user()` ข้างบนแล้ว แต่ API `/api/display/devices` ยังคืน `"defaultDisplay": ":0"`, `"outputs": []` เหมือนเดิม — proof แบบตรงที่สุด: เรียกฟังก์ชันจริงผ่าน `python3 -c` ในบริบทเดียวกับ service (`PYTHONPATH=/opt/vending-auto-setup/src`) พบว่า `find_x11_session_owner()` คืน `('first', ':0')` — **ครั้งนี้ loginctl ไม่ได้คืนค่าว่างเหมือนตอนสืบสวนรอบแรกแล้ว แต่คืน `":0"` ตรงๆ** (ค่านี้คือ display ของ GDM greeter ไม่ใช่ของ user session จริงที่ login แล้วย้ายไปรันที่ `:1`) — เรียก `_find_display_for_user('first')` ตรงๆ ในบริบทเดียวกันได้ค่า `:1` ถูกต้อง แต่ `_default_x_display()` ไม่เคยไปถึงจุดที่เรียกฟังก์ชันนี้เลย เพราะเงื่อนไขเดิม `if display: return display` เจอ `display=":0"` (non-empty = truthy) แล้ว return ค่านี้ไปก่อนทันที — สรุป root cause: **loginctl "Display" property ของ session นี้ไม่ stable** เปลี่ยนไปมาระหว่างค่าว่างกับ `:0` (ผิดทั้งคู่) ขึ้นกับจังหวะที่เรียก โค้ดเดิม trust ค่านี้ก่อนเสมอถ้าไม่ว่าง จึงพลาด
- สลับลำดับความสำคัญใน `_default_x_display()`: เรียก `_find_display_for_user()` (หา DISPLAY จริงจาก `/proc/<pid>/environ`) **ก่อนเสมอ** แล้วค่อย fallback ไปใช้ค่าจาก `loginctl` (`display`) ถ้าหาไม่เจอ แล้วค่อย `:0` เป็นทางเลือกสุดท้ายจริงๆ — กลับลำดับความไว้ใจทั้งหมด (เดิมไว้ใจ loginctl มากกว่า ตอนนี้ไว้ใจ process environ scan มากกว่า เพราะพิสูจน์แล้วว่าแม่นกว่าจริงบนเครื่องนี้)
- **ตรวจสอบ**: เจอปัญหา mount-sync ซ้ำอีกครั้ง (ตัดท้ายไฟล์ขาดกลางบรรทัด `return "OK" if status.installed else` ในฟังก์ชัน `tool_marker()` ที่อยู่คนละจุดกับที่แก้เลย) — reconstruct ด้วยวิธีเดิม (เทียบ prefix ที่สมบูรณ์กับเนื้อหาจริงจาก Read tool เขียนทับผ่าน bash ตรง) — `python3 -m py_compile` ผ่านหลัง reconstruct — `mypy --strict` บนสำเนาที่ตัด null-byte padding ออกแล้ว: ฟังก์ชันที่แก้ (`_default_x_display`) ไม่มี error ใหม่ จำนวน error รวมคงที่ 60 รายการเท่าเดิม — **ยังไม่ได้ทดสอบ end-to-end บนเครื่องจริงหลัง fix รอบนี้ (รอบที่ 3)** ต้อง deploy ใหม่ (push + `vas update`) แล้ว `sudo systemctl restart vending-auto-setup-server` แล้วเช็ค `curl -s http://127.0.0.1:8888/api/display/devices | python3 -m json.tool` ว่าได้ `"defaultDisplay": ":1"` และ `"outputs": ["Virtual1"]` หรือยัง ก่อนไปเช็คหน้าเว็บต่อ

### เพิ่ม `monitors.xml` (mutter) automation ระดับเครื่อง — แก้จอไม่หมุนตอนสลับ Kiosk user โดยไม่ต้อง trigger ผ่าน GNOME Settings บนจอเครื่องจริง
- **ที่มา**: ต่อยอดจาก `docs/kiosk-user-monitor-rotation-investigation.md` และ `docs/monitors-xml/proof-2026-07-11-option-c-system-level.md` — proof ยืนยันแล้วว่าเขียน `/etc/xdg/monitors.xml` (system-level, ไม่ผูก user) แก้ปัญหาจอ GNOME/mutter เขียนทับ rotate กลับเป็น normal ตอน kiosk-user login ได้จริง แต่รอบ proof ทำผ่านมือ (copy ไฟล์ที่ได้จากกด GNOME Settings > Displays > Apply > Keep Changes) ผู้ใช้ต้องการให้ VAS backend เป็นคนเขียนไฟล์นี้เองทั้งหมด ไม่ต้องมีขั้นตอนเข้าจอเครื่องจริงเลย
- เพิ่มโมดูลใหม่ `src/features/display/monitors_xml.py` — `MonitorsXmlManager.get_current_state()` เรียก D-Bus `org.gnome.Mutter.DisplayConfig.GetCurrentState` ตรงๆ ผ่าน `gdbus call` (แทนขั้นตอนกด Settings) โดย query `DISPLAY`/uid ของ user ที่ active session อยู่แบบ dynamic เสมอ (`find_x_session_for_user()` — ห้าม hardcode `:0`), `build_monitors_xml()` เป็น pure builder ประกอบ XML ตาม schema ที่ proof ไว้แล้ว, `write_system_level()`/`remove_system_level()` เขียน/ลบ `/etc/xdg/monitors.xml`, `user_has_own_monitors_xml()` เช็คว่า kiosk user เป้าหมายมีไฟล์ของตัวเองอยู่แล้วไหม (ถ้ามีจะชนะ system-level เสมอ ต้องเตือนไม่ใช่เงียบๆ) — `parse_get_current_state_output()` เป็น best-effort regex parser ของผลลัพธ์ `gdbus` (ยังไม่เคย proof กับ output จริงบนเครื่อง production — ดู checklist "Proof C" ที่ยังค้างใน `docs/kiosk-user-monitor-rotation-investigation.md` หัวข้อ 7)
- `server.py`: เพิ่ม `persistMonitorsXml` เข้า `/api/display/apply`, target `monitors_xml` เข้า `/api/display/persist-write`/`persist-remove`, ลบ system-level ไฟล์นี้ด้วยตอนกด Reset (`/api/display/reset`) ให้ครบเหมือน persist อื่นๆ — หน้า `/display` ส่ง `monitors_xml` status + `kiosk_target_username`/`kiosk_target_has_own_monitors_xml` เข้า template เพื่อเตือนถ้า kiosk user เป้าหมายมีไฟล์ของตัวเองอยู่แล้ว (system-level จะไม่มีผล)
- `display.html`: เพิ่มการ์ด "monitors.xml (GNOME/mutter — ระดับเครื่อง)" พร้อม toggle เขียน/ลบทันที (pattern เดียวกับการ์ด persist อื่น) + banner เตือนกรณี kiosk user เป้าหมายมีไฟล์ของตัวเองอยู่แล้ว, เพิ่มแถวใน Config Files tab ให้ดูเนื้อไฟล์จริงได้
- **ตรวจสอบ**: `python3 -m py_compile` ผ่านทั้ง `monitors_xml.py`/`display.py`/`server.py` — เจอปัญหา mount-sync แบบเดียวกับ entry ก่อนหน้าอีกครั้ง (`display.py` ค้างเป็นสำเนาของ session ก่อนหน้าทั้งไฟล์ไม่มีการแก้ไขวันนี้เลย, `server.py` ตัดท้ายไฟล์ขาดหายกลางบรรทัดใน docstring ของ `_find_display_for_user`) — reconstruct ทั้งสองไฟล์จากเนื้อหาที่ยืนยันถูกต้องผ่าน Read tool (เขียนสำเนาใหม่ผ่าน `outputs/` แล้ว `cp`/splice ทับ path จริงในแซนด์บ็อกซ์โดยตรง ไม่ผ่าน Edit tool ซ้ำ) — `mypy --strict --cache-dir=/tmp` (ต้องระบุ cache dir แยกเพราะ mount นี้ทำ SQLite lock ไม่ได้ ชน `disk I/O error`): `monitors_xml.py` ไม่มี error เลย, `display.py`/`kiosk/manager.py` มี error ที่มีอยู่ก่อนแล้ว (unused type:ignore, dict variance ของ `normalize_chrome_flags`) แต่ไม่ใช่จุดที่แก้วันนี้, `server.py` รวม 60 error เท่าเดิมกับ baseline ที่บันทึกไว้ใน entry ก่อนหน้า ไม่มี error ใหม่จุดไหนที่แก้วันนี้เลย — **ยังไม่ได้ verify jinja2 syntax ของ `display.html` แบบอัตโนมัติ** (ไฟล์ template ทั้งสองค้างเป็นสำเนาเก่าใน mount เหมือนกัน reconstruct เต็มไฟล์ไม่คุ้มเวลาในรอบนี้ ตรวจด้วยสายตาแทนว่า `{% if %}/{% elif %}/{% endif %}` และ div เปิด-ปิดครบ) — **ยังไม่ได้ proof/deploy บนเครื่องจริง** ต้องปิด checklist "Proof C" (D-Bus introspect) ก่อน แล้วทดสอบ `persistMonitorsXml` end-to-end บนเครื่อง production จริงก่อนใช้งาน

### แก้จอเทา/ดำ + Chromium client-side exception ตอนใช้ Openbox session
- **ที่มา**: ผู้ใช้แจ้งว่าตั้ง Kiosk user ให้ใช้ Openbox (เบากว่า GNOME) แล้วเจอจอเทาทะมึนตอน boot บางครั้ง และบางครั้งเจอ Chromium ขึ้นหน้า "Application error: a client-side exception has occurred" ของเว็บแอป kiosk เอง — วิเคราะห์ได้ 3 สาเหตุ: (1) Openbox ไม่มี compositor/background painting ของตัวเองเหมือน GNOME เห็นสีเทาดิบของ X root window ตรงๆ (2) autostart ของ Openbox ยิง chromium ทันทีไม่รอเครือข่ายพร้อมเหมือน GNOME session ทำให้เว็บแอปฝั่ง client fetch พังตอนบูตเร็วกว่าที่เน็ตจะมา (3) chromium อาจเปิด fullscreen ก่อนจอหมุนเสร็จ (`display-session.sh` มี delay 5 วิ + poll แต่ autostart ของ Openbox ไม่รอ)
- เพิ่ม `DISPLAY_READY_MARKER_NAME`/`display_ready_marker_path()` (`src/features/display/display.py`) — `display-session.sh` (`build_display_session_script`) ลบ marker เก่าทิ้งตอนเริ่มรัน แล้ว touch ไฟล์นี้เฉพาะตอนหมุนจอ+ปรับ touch สำเร็จจริงเท่านั้น — ลบ marker ด้วยตอน `remove_session_persist()`/`reset_touch_mapping()` ให้สอดคล้องกับไฟล์ persist อื่น
- เพิ่ม `build_openbox_autostart_preamble()` (`src/features/kiosk/manager.py`) — ใส่ไว้หลัง shebang ของสคริปต์ autostart เฉพาะ session แบบ Openbox เท่านั้น (`_insert_after_shebang()`, ผ่าน `write_autostart()`): `xsetroot -solid "#000000"` กันจอเทาดิบ, รอไฟล์ marker ข้างบน (bounded, default 10 วิ) กันจอหมุนไม่ทันก่อน chromium เปิด, รอ `getent hosts <hostname ของ URL>` (bounded, default 20 วิ) กันเว็บแอป client-side exception ตอนเน็ตยังไม่พร้อม
- เพิ่ม auto-reload watchdog: `build_kiosk_launch_script()` รับ `auto_reload_minutes` ใหม่ — ถ้า > 0 และเปิด restart loop อยู่ด้วย จะ wrap คำสั่ง chromium ด้วย `timeout <นาที*60>s` ให้ restart loop เดิมเปิดใหม่ให้เป็นระยะ แก้เคส client-side exception ที่ไม่ทำให้ chromium process ตาย (restart-on-exit เดิมช่วยไม่ได้) — `_parse_kiosk_script()` ตัด Openbox preamble ออกก่อนเสมอถึงจะ parse ค่า `sleep`/`timeout` กลับมา กัน regex ไปจับ `sleep 1` ของ preamble แทนค่าจริง — เพิ่ม field นี้เข้า `KioskAutostartStatus`
- `server.py`: `/api/kiosk/autostart` รับ `auto_reload_minutes` ส่งต่อ `write_autostart()` + บันทึก audit/config log, `_kiosk_page_context()` ส่งค่านี้เข้า script preview — `kiosk.html`: เพิ่มช่อง "รีโหลดเบราว์เซอร์อัตโนมัติทุก (นาที)" ในแท็บ "เปิดแอปอัตโนมัติ" ผูกกับ preview + save function
- **ตรวจสอบ**: เหมือน entry ก่อนหน้า — เจอ mount-sync ค้าง (`kiosk/manager.py` เป็นสำเนา session ก่อนหน้า) reconstruct ผ่าน `outputs/` แล้ว `cp` ทับ, `python3 -m py_compile` ผ่าน, `mypy --strict --cache-dir=/tmp` ไม่มี error ใหม่จากจุดที่แก้ (error เดียวที่เจอในไฟล์นี้ — dict variance ของ `normalize_chrome_flags` — มีอยู่ก่อนแก้แล้ว ไม่เกี่ยวกับโค้ดใหม่) — **ยังไม่ได้ทดสอบ end-to-end บนเครื่อง Openbox จริง** ต้อง deploy แล้วสลับ session type เป็น Openbox ทดสอบ reboot ซ้ำหลายรอบเพื่อเช็คว่าจอเทา/error หายจริงไหม ก่อนถือว่าปิดเคสนี้

### แก้ error message ของ toggle "Persist monitors.xml" ที่ generic เกินไปจนเดาสาเหตุ D-Bus ล้มเหลวไม่ออก
- **ที่มา**: ผู้ใช้กด toggle "monitors.xml (GNOME/mutter — ระดับเครื่อง)" ในหน้า `/display` แล้วขึ้น error "อ่านค่าฮาร์ดแวร์จอผ่าน D-Bus (mutter) ไม่สำเร็จ — เช็คว่า session ตอนนี้เป็น GNOME/mutter จริงไหม" ซึ่งเป็นแค่ generic message ไม่บอกสาเหตุจริง — พบว่า `MonitorsXmlManager.get_current_state()` เดิมทิ้ง stderr/stdout ของ `gdbus call` ทิ้งทั้งหมดแล้วคืนแค่ `None` ทำให้แยกไม่ออกว่าล้มเหลวเพราะ (1) kiosk user ที่ login อยู่ตอนนี้ตั้ง session เป็น Openbox (ไม่มี mutter/GNOME Shell ให้เรียก D-Bus เลย — เคสที่พบจริงกับผู้ใช้ เพราะเพิ่งสลับ kiosk-user ไปทดสอบ Openbox ก่อนหน้านี้ในวันเดียวกัน) (2) D-Bus service ยังไม่พร้อม/timeout จริง หรือ (3) `parse_get_current_state_output()` (best-effort regex parser ที่ยังไม่เคย proof กับ output จริง) parse ไม่ได้
- `src/features/display/monitors_xml.py::get_current_state()`: เปลี่ยน return type เป็น `tuple[MonitorState | None, str | None]` คืน error detail จริงกลับมาด้วยเสมอ — กรณี `gdbus` exit code ไม่ใช่ 0 คืน stderr/stdout จริง, กรณี parse ไม่ได้ คืนข้อความบอกว่า parser เป็น best-effort พร้อม raw output 300 ตัวอักษรแรก
- `src/server.py::_write_system_monitors_xml()`: เพิ่มเช็ค `collect_accounts_service_status(query_user).session_type` **ก่อน** ยิง D-Bus เลย — ถ้าเป็น `"openbox"` คืน error message เฉพาะเจาะจงทันทีว่า session ปัจจุบันเป็น Openbox ไม่มี mutter ให้เรียก พร้อมบอกวิธีแก้ (สลับ kiosk user เป็น GNOME ชั่วคราวก่อน) แทนที่จะปล่อยให้ไปเจอ error generic จาก D-Bus ตอนหลัง — ต่อท้าย error message เดิมด้วย `error_detail` จริงถ้าไปถึงขั้น D-Bus แล้วยังล้มเหลว
- **ตรวจสอบ**: เจอปัญหา mount-sync ค้างอีกครั้งกับทั้งสองไฟล์ (`monitors_xml.py` ค้าง 276 บรรทัดแทนที่จะเป็น 309, `server.py` ตัดขาดกลางสตริง `if name and name != "ro` ที่บรรทัด 3902) — reconstruct `monitors_xml.py` ทั้งไฟล์ผ่าน `outputs/` แล้ว `cp` ทับ, ส่วน `server.py` ใช้วิธี byte-splice เดิม (หา anchor `def _desktop_user() -> str | None:` ที่ไม่ซ้ำ ตัด prefix ที่ถูกต้องอยู่แล้วไว้ แล้วต่อ tail ที่ verify แล้วจาก Read tool เข้าไปแทน) — `python3 -m py_compile` ผ่านทั้งสองไฟล์, grep-count ฟังก์ชัน/คลาสสำคัญทั้งหมดในไฟล์ที่ splice ได้ 1 รายการเท่านั้น (ไม่มีซ้ำ), `mypy --strict --cache-dir=/tmp/mypy_cache`: `monitors_xml.py` ไม่มี error เลย, `server.py` รวม 60 error เท่ากับ baseline เดิมทุกประการ ไม่มี error ใหม่ในจุดที่แก้ — **ยังไม่ได้ทดสอบบนเครื่องจริงว่า error message ใหม่ขึ้นถูกต้องจริงตอน session เป็น Openbox** (เขียนตาม logic ที่อ่านจากโค้ด `collect_accounts_service_status` เท่านั้น) ต้องกดปุ่มนี้ซ้ำบนเครื่องจริงเพื่อยืนยันว่าได้ error message ใหม่ที่เจาะจงกว่าเดิม

### ปิดใช้งาน toggle "Persist monitors.xml" ล่วงหน้า + แสดงคำเตือนตั้งแต่โหลดหน้า ถ้า kiosk user เป็น Openbox
- **ที่มา**: ผู้ใช้ขอให้ทำ state ให้ถูกต้องตั้งแต่แรกแทนที่จะปล่อยให้กดแล้วเจอ error — ถ้า kiosk user ตอนนี้ตั้ง session เป็น Openbox ให้ปิดปุ่มไปเลยพร้อมโชว์คำเตือน แทนที่จะให้กดแล้วค่อยรู้จาก error message (entry ก่อนหน้าในวันเดียวกัน)
- `src/server.py::display_settings()` (route `/display`): เพิ่มการคำนวณ `kiosk_target_session_type` จาก `collect_accounts_service_status(kiosk_target_username).session_type` (ฟังก์ชันเดียวกับที่ `_write_system_monitors_xml()` ใช้เช็คตอน submit) ส่งเข้า template — default เป็น `"gnome"` ถ้าไม่มี kiosk user เป้าหมายเลย
- `src/web/templates/display.html`: การ์ด monitors.xml เพิ่มเงื่อนไข `{% if kiosk_target_session_type == "openbox" %}` เป็นลำดับแรกสุด (ตัดหน้าคำเตือนอื่นๆ ที่มีอยู่แล้ว) โชว์ banner สีแดง (`bg-danger`) อธิบายว่า Openbox ไม่มี mutter ให้ใช้ D-Bus พร้อมวิธีแก้ — ปิด `<input id="persistMonitorsXml">` ด้วย `disabled` attribute และใส่ `opacity-50 cursor-not-allowed` + `title` tooltip บน `<label class="toggle-switch">` เมื่อเข้าเงื่อนไขนี้ (input ที่ `disabled` เบราว์เซอร์จะไม่ยิง `change` event ให้ JS เลย จึงไม่ต้องแก้ `bindPersistToggle()` เพิ่ม)
- **ตรวจสอบ**: เจอ mount-sync ค้างอีกรอบทั้ง `server.py` (ตัดขาดกลางคำ `_is_display_c` เพราะบรรทัดขยับจากการเพิ่มโค้ดใหม่) และ `display.html` (ค้าง 977 บรรทัดแทนที่จะเป็น 1132 บรรทัดจริง) — reconstruct `server.py` ด้วย byte-splice เดิม (anchor เดิม `def _desktop_user`), reconstruct `display.html` ทั้งไฟล์ผ่าน `outputs/` แล้ว `cp` ทับ (ไฟล์ยาวเกิน limit เดียวของ Read tool เลยอ่านสองรอบ 1-818 และ 819-1133 ประกอบกันแทน) — `python3 -m py_compile` ผ่าน, ตรวจ syntax ของ Jinja template ด้วย `jinja2.Environment().parse()` ตรงๆ ในแซนด์บ็อกซ์ (เจอ error "Unexpected end of template" ตอนไฟล์ยังค้างอยู่ พอ reconstruct แล้ว parse ผ่านสะอาด) — `mypy --strict --cache-dir=/tmp/mypy_cache` รวม 60 error เท่าเดิมกับ baseline ไม่มี error ใหม่ — **ยังไม่ได้เปิดหน้าเว็บจริงดูว่า banner/disabled แสดงผลถูกต้องตามที่ตั้งใจ** (ตรวจแค่ syntax level ผ่าน jinja2 parser เท่านั้น ยังไม่ได้ render จริงด้วยค่า context จริง) ต้องเปิดหน้า /display บนเครื่องจริง (ที่ kiosk user เป็น Openbox อยู่ตอนนี้) เพื่อยืนยันว่า toggle ถูกปิดจริงและ banner สีแดงขึ้นถูกต้อง

### แก้ `parse_get_current_state_output()` parse raw output จริงจาก gdbus ไม่ได้ — ปิด checklist "Proof C" ที่ค้างมาตั้งแต่ต้น
- **ที่มา**: หลัง fix สอง entry ก่อนหน้าในวันเดียวกัน (แก้ session เป็น GNOME แล้ว + toggle เปิดใช้งานได้แล้ว) ผู้ใช้กด toggle จริงบนเครื่อง production ได้ error ใหม่: "เรียก D-Bus สำเร็จ (gdbus exit code 0) แต่ parse ผลลัพธ์ไม่ได้" พร้อม raw output ที่ error message โชว์ออกมา (ผลจาก fix ใน entry ก่อนหน้าที่เพิ่ม raw output ไว้ใน error message เอง) — เห็น raw output จริงครั้งแรกแล้วพบว่า parser เดิมสันนิษฐานผิด: คิดว่าตัวเลขใน mode tuple จะมี type annotation กำกับเสมอ (`uint32 800, uint32 600, double 60.0`) แต่ output จริงจาก `gdbus call` ไม่ print annotation ซ้ำสำหรับสมาชิกที่ type เดียวกันในโครงสร้างซ้อน ได้ตัวเลขดิบๆ ล้วน (`('800x600@60', 800, 600, 60.0, 1.0, [1.0], {'is-preferred': <true>})`) — เป็น checklist "Proof C" ที่เอกสาร `docs/kiosk-user-monitor-rotation-investigation.md` ระบุไว้ตั้งแต่แรกว่ายังไม่เคย proof กับเครื่องจริง ตอนนี้ปิด checklist นี้ได้แล้ว
- `src/features/display/monitors_xml.py::parse_get_current_state_output()`: เขียน regex ของ mode entry ใหม่ทั้งหมดให้ตรงกับ shape จริง `('<mode_id>', <width>, <height>, <rate>, <preferred_scale>, [<supported_scales>], {<properties>})` โดยไม่พึ่ง "uint32"/"double" annotation อีกต่อไป — วน `findall()` หา mode ทุกตัวแล้วเลือกตัวที่มี `'is-current': <true>` อยู่ใน properties, ถ้าไม่เจอเลยสักตัว fallback ไปใช้ mode แรกเหมือนเดิม
- **ตรวจสอบ**: เขียน unit test เฉพาะกิจ (ไม่ได้เก็บเป็นไฟล์ถาวร) รัน parser กับ 3 เคส: (1) raw output จริงที่ผู้ใช้ส่งมาแบบตัดที่ 300 ตัวอักษร (เหมือนใน error message) → ได้ `800x600` ถูกต้อง (2) เคสจำลองที่มี `'is-current': <true>` อยู่ที่ mode ตัวที่ 2 → เลือกตัวที่มี flag ถูกต้อง ไม่ใช่ตัวแรก (3) เคสไม่มี flag `is-current` เลยสักตัว → fallback ไปตัวแรกตามที่ตั้งใจ — เจอ mount-sync ค้างอีกครั้งระหว่างแก้ (ตัด mypy fix รอบสองขาดกลางบรรทัด `height=`) reconstruct ด้วย byte-splice ตามเดิม — `python3 -m py_compile` ผ่าน — `mypy --strict --cache-dir=/tmp/mypy_cache`: พบ error ใหม่ 2 รายการจากการที่ mypy narrow type `width`/`height` ไม่ได้ข้าม for-loop (แก้โดยเปลี่ยนไปใช้ตัวแปร `selected: tuple[str,str,str] | None` ตัวเดียวแทน) แก้แล้วกลับมาที่ 60 error เท่า baseline เดิม ไม่มี error ใหม่เหลือ — **ยังไม่ได้ทดสอบกับ raw output แบบเต็ม (ไม่ตัด 300 ตัวอักษร) จากเครื่องจริง** เพราะ error message ที่ผู้ใช้ส่งมาถูกตัดไว้ที่ 300 ตัวอักษรอยู่แล้ว ต้องกด toggle ซ้ำบนเครื่องจริงเพื่อยืนยันว่าเขียนไฟล์ `/etc/xdg/monitors.xml` สำเร็จจริงรอบนี้

### รวม `APP_VERSION` ให้อ่านจาก `pyproject.toml` ที่เดียว (single source of truth)
- **ที่มา**: `APP_VERSION` (`src/core/config.py`) hardcode แยกจาก `version` ใน `pyproject.toml` เป็นคนละค่ากันมาตลอด (บังเอิญเท่ากันคือ `"0.1.0"`) ทำให้ต้องแก้ 2 ที่เวลาจะ bump version ไม่งั้นค่าที่แสดงในหน้า "อัปเดตระบบ"/`vas --version`/version compare ตอนเช็ค release จะไม่ตรงกับที่ประกาศใน package metadata
- เปลี่ยน `APP_VERSION` ให้เรียก `_read_pyproject_version()` ตอน import — อ่านค่า `version = "..."` จากใต้ `[project]` section ของ `pyproject.toml` ด้วย regex ล้วนๆ (ไม่ใช้ `tomllib` เพราะเป็นของ Python 3.11+ แต่เครื่องเป้าหมาย Ubuntu 22.04 jammy มากับ Python 3.10 เป็นค่าเริ่มต้น) — resolve path แบบ `Path(__file__).resolve().parents[2] / "pyproject.toml"` (ใช้ได้ทั้งตอน dev ในนี้ และตอนติดตั้งจริงที่ `/opt/vending-auto-setup` เพราะ `shutil.copytree(source_dir, install_dir)` ใน `services/updater.py` ก็อปปี้ทั้ง repo root รวม `pyproject.toml` ไปด้วยอยู่แล้ว ไม่ใช่แค่ `src/`) — ถ้าอ่านไม่ได้ (path ไม่มี/ไฟล์เพี้ยน) fallback กลับไปที่ `_FALLBACK_VERSION = "0.1.0"` ไม่ crash
- **ผลลัพธ์กับผู้ใช้**: ต่อไปนี้แก้ version ที่เดียวพอ — เปลี่ยน `version = "x.y.z"` ใน `pyproject.toml` แล้วหน้า "อัปเดตระบบ", `vas --version`/`vas version`, และ logic เทียบเวอร์ชั่นตอนเช็ค release จะอ่านค่าใหม่ตรงกันทันที (แค่ต้อง restart service ให้ Python re-import module)
- ตรวจสอบ: คัดลอกเนื้อหาไฟล์จริงจาก Read tool ไปรัน `python3 -m py_compile`/`ast.parse` แยกใน sandbox ผ่านทั้งคู่ (bash mount ของไฟล์นี้ในเซสชันนี้ยังไม่ทันเช็คว่าค้างหรือไม่ ใช้แนวทางเดิมเพื่อความชัวร์) — ทดสอบ logic การ parse แยกด้วยไฟล์ `pyproject.toml` จริงในโปรเจกต์ ได้ค่า `"0.1.0"` ตรงตามที่ประกาศไว้ — ยังไม่ได้รัน `pytest`/`mypy --strict` เต็มโปรเจกต์ (ต้องมี env ครบ), ยังไม่ได้ทดสอบ path resolution บนเครื่อง Ubuntu จริงหลัง self-update จริง (เคสที่ก็อปปี้ไปที่ `/opt/vending-auto-setup`) ควร verify อีกครั้งหลัง deploy

### หน้า "อัปเดตระบบ" — เพิ่มโหมด Dev สำหรับดึง source ล่าสุดจาก branch มาติดตั้งทันที (ไม่ผ่าน GitHub release)
- **ที่มา**: หน้า Update เดิมใช้ GitHub Releases เป็นเกณฑ์เดียวในการเช็ค/แสดงปุ่มอัปเดต (`check_latest_release()` เรียก `/releases/latest`) แต่ repo ของผู้ใช้ยังไม่เคยสร้าง release เลย ("ไม่พบ release บน GitHub repository นี้") ทำให้กดอัปเดตจริงไม่ได้แม้ push code ใหม่ขึ้น branch ไปแล้ว ทั้งที่กลไก `SelfUpdater`/`start_web_update()` (`src/services/updater.py`) ที่ใช้ติดตั้งจริงรองรับการดึง tarball จาก branch HEAD อยู่แล้ว (`archive_url()` เมื่อ `version="latest"` จะดึงจาก `refs/heads/{branch}.tar.gz` ตรงๆ ไม่ต้องมี release/tag) เพียงแต่ยังไม่มีทางเลือกใน UI ให้เลือก branch หรือข้ามการเช็ค release
- เพิ่ม query param `?branch=<name>` ให้ `/api/update/stream` (`src/server.py`) — ส่งต่อเป็น `start_web_update(branch=branch)` โดยไม่ผ่าน `check_latest_release()` เลย (endpoint เดิมเริ่มอัปเดตทันทีอยู่แล้วโดยไม่รอผลเช็ค แค่ไม่เคยรับ branch จาก client) — เพิ่ม `is_valid_branch_name()`/`BRANCH_NAME_RE` ใน `updater.py` เช็คชื่อ branch (`^[A-Za-z0-9._\-/]+$`, ไม่ขึ้นต้นด้วย `/`/`-`, ไม่มี `..`) ก่อนนำไปประกอบ URL ป้องกัน path/URL injection จาก input ที่ผู้ใช้กรอกเอง — ถ้าไม่ผ่าน validation คืน error ทันทีไม่เริ่ม thread
- เพิ่มการ์ด "โหมด Dev — ดึงจาก Source โดยตรง" ใน `update.html` ระหว่างการ์ดเวอร์ชั่นกับ progress card — ช่อง input ให้กรอกชื่อ branch เอง (default `main` ถ้าเว้นว่าง) + ปุ่ม "ดึง Source ล่าสุด & ติดตั้ง" ที่เรียก flow เดิมทันทีไม่เช็คเวอร์ชั่นก่อน (ตามที่ผู้ใช้ยืนยัน — ใช้สำหรับทดสอบระหว่าง develop เท่านั้น) — refactor JS: แยก `startUpdate()` เดิมออกเป็น `beginUpdateFlow(branch, disableBtn)` ใช้ร่วมกัน แล้วเพิ่ม `window.startDevUpdate()` อ่านค่าจาก `#dev-branch-input` ต่อเป็น query string ของ `EventSource` — ปุ่ม/state/terminal log ใช้ progress card เดิมร่วมกันทั้ง 2 โหมด ไม่ต้องสร้างซ้ำ
- **พบและแก้ไฟล์ `src/server.py` ที่เสียหายอยู่ก่อนแล้ว (ไม่เกี่ยวกับงานนี้โดยตรง แต่บล็อก syntax check)**: ท้ายไฟล์มี fragment ของ `def vpn_connection_label(...)` ซ้ำอยู่ 2 ชุดที่เขียนไม่จบ (ชุดแรกขาดหายกลางชื่อพารามิเตอร์ ชุดที่สองเป็นโค้ด logic คนละแบบกับตัวจริงที่ใช้งานอยู่จริงที่บรรทัด 249 ซึ่ง register เป็น jinja global ไปแล้ว) — ลบ fragment ที่ค้างทั้ง 2 ชุดออก ยืนยันด้วย `python3 -m py_compile` ผ่าน ไม่กระทบฟังก์ชัน `vpn_connection_label` ตัวจริงที่ใช้อยู่ในหน้า WireGuard
- **เพิ่ม auto-restart หลังอัปเดตเสร็จ**: เดิม `done-bar` โชว์ปุ่ม "รีสตาร์ท" ให้ผู้ใช้กดเองหลังอัปเดตสำเร็จ (ทั้ง release mode และ dev mode) — เปลี่ยนให้ SSE event `done` เรียก `restartServer()` เอง (ยิง `POST /api/server/restart` ที่มีอยู่แล้ว → `systemctl restart vending-auto-setup-server.service` หลัง delay 0.5 วิ) ทันทีที่อัปเดตเสร็จโดยไม่ต้องรอผู้ใช้กด — host/port ยังคงเดิมเสมออยู่แล้วเพราะ systemd unit อ่านจาก `EnvironmentFile=/etc/default/vending-auto-setup-server` (`VAS_SERVER_HOST`/`VAS_SERVER_PORT`) ที่ตั้งไว้ตอน install ไม่เกี่ยวกับ process ที่ restart เลย — เพิ่ม guard `_restartTriggered` กัน fetch ซ้ำถ้าผู้ใช้ยังกดปุ่ม "รีสตาร์ทตอนนี้" (คงไว้เป็น fallback) หลัง auto-fire ไปแล้ว — เปลี่ยนข้อความ/ไอคอนใน done-bar จาก static เป็น dynamic (spinner → check) สะท้อนสถานะ "กำลังรีสตาร์ท" / "รีสตาร์ทแล้ว รอเซิร์ฟเวอร์กลับมา"
- ตรวจสอบ: bash sandbox mount ของทั้ง `server.py`/`updater.py` ค้างเหมือนที่เจอซ้ำๆ ในหลาย entry ก่อนหน้า (แสดงเนื้อหาเก่า/มี null byte padding ท้ายไฟล์หลัง edit) — คัดลอกเนื้อหาไฟล์จริงจาก Read tool ไปสร้างสำเนาแยกใน sandbox ก่อนรัน `python3 -m py_compile` และ `ast.parse` จนผ่านทั้งคู่ — แยก JS block ของ `update.html` (รวมส่วน auto-restart ที่เพิ่มรอบหลัง) ไปรัน `node --check` ผ่านทุกครั้งที่แก้ — mypy แบบ standalone (ไม่มี project context) ขึ้น internal error ของตัวเครื่องมือเอง ไม่ใช่จาก code จึงข้าม ใช้ py_compile + ตรวจ type hint ด้วยตาแทน — ยังไม่ได้ทดสอบ end-to-end บนเครื่อง Ubuntu จริง (ยิงอัปเดตจริงจาก branch ทดสอบ แล้วเช็คว่า `/opt/vending-auto-setup` อัปเดตตรงตาม commit ล่าสุด, service restart แล้ว listen พอร์ตเดิมจริง, และ SSE connection ไม่หลุดก่อน restart request ยิงออกไปสำเร็จ) ควร verify อีกครั้งหลัง deploy

### หน้า "จอแสดงผล" — Reactive protocol switch, แก้บั๊ก WaylandEnable comment สะสม, hover state ของ Tab menu
- **Reactive protocol switch**: กด "เปลี่ยนมาใช้ X11/Wayland" ในแท็บ "Display Protocol" เดิม `setTimeout(() => location.reload(), 1200)` reload หน้าทั้งหน้าเสมอหลัง API สำเร็จ ทำให้กะพริบ/เสีย scroll position — แก้โดยเพิ่ม id ให้ badge "Active"/กล่อง "กำลังใช้งาน"/ปุ่มสลับของทั้ง 2 การ์ด (`x11ActiveBadge`/`x11ActiveBox`/`x11SwitchBtn`, `waylandActiveBadge`/`waylandActiveBox`/`waylandSwitchBtn`) แล้วเขียน `renderProtocolState(disabled)` ใน `display.html` อัปเดต class `active`/`hidden` ของทั้ง 2 การ์ดตรงๆ จาก `gdmWayland.disabled` ที่ endpoint `/api/display/wayland` คืนมาอยู่แล้ว โดยไม่ reload หน้า — บรรทัด "ใช้งานได้หลัง re-login" ของ session banner ด้านบนยังคงเดิม (สะท้อน session จริงที่ต้อง re-login ถึงจะเปลี่ยน ไม่ใช่ config ที่อัปเดตทันที)
- **บั๊ก `WaylandEnable=false` สะสมหลายบรรทัดใน `/etc/gdm3/custom.conf`**: `build_gdm_wayland_config()` (`src/features/display/display.py`) เดิมมองหาเฉพาะบรรทัด active (ไม่ comment) ด้วย `_is_active_ini_key()` เวลา toggle กลับไป X11 ถ้าบรรทัดเดิมถูก comment ไว้จากรอบก่อน (ไม่ match เพราะขึ้นต้นด้วย `#`) โค้ดจะ `insert` บรรทัดใหม่เพิ่มเข้าไปโดยไม่ลบของเก่าทิ้ง สลับไปมาหลายรอบจึงเห็น `#WaylandEnable=false` ซ้ำกันหลายบรรทัดใน Config Files tab — แก้โดยเพิ่ม param `allow_commented` ให้ `_is_active_ini_key()` (strip `#`/`;` ออกก่อนเช็ค key) แล้วเปลี่ยน `build_gdm_wayland_config()` ให้หาทุกบรรทัดที่ match key (comment หรือไม่ก็ตาม), แทนที่บรรทัดแรกด้วยค่าใหม่ แล้วลบบรรทัดซ้ำที่เหลือทั้งหมด — self-heal ไฟล์ config เดิมที่พังจาก bug นี้อยู่แล้วได้ทันทีที่ toggle ครั้งถัดไป
- **Hover state ของ Tab menu หน้า Display ไม่ทำงาน**: ปุ่ม tab (`.disp-tab-btn`) ที่ active ตอน initial render (`loop.first`) ไม่มี class `hover:*` ติดมาด้วยเลยตั้งแต่แรก (template ใส่ class คนละชุดให้ active/inactive) แล้ว `switchTab()` เดิมใช้ `classList.toggle()` แค่บาง class (`bg-accent/8`/`text-accent`/`font-semibold`/`text-muted`) ไม่เคยจัดการ hover class — พอสลับปุ่มนั้นออกจาก active จึงไม่มี hover state ตลอดไป — แก้โดยเปลี่ยน hover class ให้ตรงกับ convention (`hover:bg-surface` แทน `hover:bg-line/6`) และ rewrite `switchTab()` ให้ strip + append class ทั้งชุดทุกครั้งเหมือนหน้า "คีออส" (`setTab()`) แทนการ toggle ทีละ class — เพิ่ม section "JS Tab Switch Pattern" ใน `INSTRUCTIONS.md` ให้เป็น convention บังคับสำหรับหน้าที่มี tab menu ทุกหน้า
- ตรวจสอบ: เพิ่ม unit test 2 เคสใน `tests/test_display.py` (`test_gdm_wayland_collapses_duplicate_commented_lines`, `test_gdm_wayland_toggle_twice_does_not_accumulate_lines`) ครอบคลุมทั้งเคส config ที่พังอยู่แล้ว (มีบรรทัดซ้ำ 5 บรรทัด) และเคส toggle สลับไปมา 5 รอบ — รัน `pytest tests/test_display.py` ผ่านทั้งหมด 11 เคส (รวมเทสต์เดิม), `ruff check` ผ่าน, `mypy --config-file pyproject.toml` ไม่มี error ใหม่จากจุดที่แก้ (2 warning ที่เจอเป็นของเดิมใน `_chown_to_effective_user` ไม่เกี่ยวกับรอบนี้ ยืนยันด้วยการรัน mypy กับไฟล์ต้นฉบับก่อนแก้เทียบกัน) — bash sandbox mount ของไฟล์ที่แก้ผ่าน Edit tool ค้างเหมือนที่เคยพบมาก่อน (ดู entry ก่อนหน้า) จึงคัดลอกเนื้อหาไฟล์จริงจาก Read tool ไป sync ใน sandbox แยกก่อนรันเทสต์ — ยังไม่ได้ทดสอบบนเครื่อง Ubuntu จริงกับ GDM/`/etc/gdm3/custom.conf` จริง ควร verify อีกครั้งหลัง deploy

### WireGuard — แก้กับดัก sudo/non-sudo ที่ทำให้ store dir ค้างเป็นของ root (auto-chown คืน user)
- **ปัญหาที่รายงาน**: ผู้ใช้สับสนลำดับการใช้ `sudo` กับ `vas wireguard` ตอนติดตั้งเครื่องครั้งแรก แล้วเจอ Permission denied ตอนรันคำสั่งที่ไม่ได้ใส่ sudo
- **สาเหตุจริง (ไม่ใช่แค่ความเข้าใจผิดของผู้ใช้)**: มีแค่ 3 คำสั่งที่บังคับ root ผ่าน `require_root()` (`cli.py`) คือ `install`/`sync`/`unsync` ส่วน `init-config`/`validate`/`save`/`status`/`history`/`show` ไม่บังคับ แต่ก็ไม่ได้ป้องกันไม่ให้รันด้วย sudo — `default_store_dir()` (`src/features/wireguard/manager.py`) resolve path ผ่าน `SUDO_USER`/`HOME` ไปยังโฮมของ user จริงเสมอ แต่ไฟล์ที่เขียนจะเป็นเจ้าของโดย root ถ้า process รันเป็น root ซ้ำร้ายกว่านั้น web dashboard (`src/services/server_service.py` — systemd unit ไม่มี `User=` จึงรันเป็น root เสมอ) เรียก `WireGuardManager` ตรงๆ ผ่าน `server.py` โดยไม่ผ่าน `require_root()` เลย ทำให้ทุกครั้งที่ใช้หน้าเว็บ WireGuard (Save/Template ฯลฯ) ไฟล์ใน `~/.config/vending-auto-setup/wireguard/` กลายเป็นของ root โดยอัตโนมัติอยู่แล้ว แล้วครั้งถัดไปที่รัน `vas wireguard save`/`show`/`history` แบบไม่ sudo จะเจอ Permission denied ทันทีเพราะ `chmod_private()` ล็อกไฟล์เป็น `600`
- แก้โดยเพิ่ม `reclaim_ownership()`/`_resolve_home()` ใน `manager.py` — เมื่อ `os.geteuid() == 0` จะ chown ไฟล์/โฟลเดอร์ที่เพิ่งเขียนกลับเป็นเจ้าของของโฮมจริง (ดู owner จาก `os.stat(home)` แทนที่จะพึ่ง `SUDO_USER` อย่างเดียว เพราะ systemd service ไม่มี `SUDO_USER` มีแค่ `HOME`) เรียกใช้ท้าย `init_config()`/`save()`/`sync()`/`unsync()` — เจาะจงเฉพาะไฟล์ใน store dir (`~/.config/...`) เท่านั้น ไม่แตะ active config ใต้ `/etc/wireguard` ซึ่งต้องเป็นของ root ตามปกติ — ไม่ต้องบังคับ sudo เพิ่มในคำสั่งที่ไม่จำเป็น (`validate`/`history`/`show`/`status` ยังใช้แบบไม่ sudo ได้ตามเดิม)

### ติดตั้ง Node.js/AnyDesk ผ่านหน้าเว็บล้มเหลว — gpg คุย /dev/tty ไม่ได้
- Test บนเครื่องจริงพบว่าติดตั้ง Node.js และ AnyDesk ผ่านหน้า "โปรแกรมเพิ่มเติม" ค้างที่ขั้น `curl ... | gpg --dearmor -o ...` เสมอ ด้วย error `gpg: cannot open '/dev/tty': No such device or address` ตามด้วย `curl: (23) Failed writing body` (curl เขียนเข้า pipe ที่ gpg ปิดไปแล้วไม่ได้) — สาเหตุคือคำสั่งรันผ่าน `subprocess.Popen` จาก Flask backend (ไม่มี controlling terminal) แต่ `gpg --dearmor` พยายามเปิด `/dev/tty` เพื่อโต้ตอบ (เช่น ถามยืนยันถ้าไฟล์ปลายทางมีอยู่แล้ว) ตามค่าเริ่มต้นเมื่อไม่ได้สั่ง batch mode
- แก้โดยเพิ่ม `--batch --yes --no-tty` ให้ `gpg --dearmor` ทุกจุดที่เรียกผ่าน `bash -lc` (`src/features/packages/settings.py`: node, anydesk — ทั้งฝั่ง web install ที่เจอบั๊ก และ `src/features/packages/installers.py`: node, anydesk ที่เป็น CLI equivalent เดิม ก็มีบั๊กเดียวกันแฝงอยู่ แก้ไปพร้อมกันเพื่อความสอดคล้อง) — `--batch --no-tty` ปิดการโต้ตอบผ่าน terminal ทั้งหมด `--yes` ให้เขียนทับไฟล์ keyring เดิมได้โดยไม่ถาม
- Docker ไม่กระทบ เพราะขั้นตอนดาวน์โหลด key ของ Docker ใช้ `curl -o` เขียนไฟล์ `.asc` ตรงๆ ไม่ได้ผ่าน `gpg --dearmor`

### หน้า "โปรแกรมเพิ่มเติม" — แก้บั๊ก onclick พัง (JSON.stringify ใน double-quoted attribute)
- ปุ่ม `installPkg`/`uninstallPkg`/`openSubSelectModal` เดิมสร้าง `onclick="fn(' + JSON.stringify(p.id) + ')"` — `JSON.stringify` คืน string ที่ครอบด้วย double quote (`"node"`) ไปฝังอยู่ใน HTML attribute ที่ครอบด้วย double quote เหมือนกัน ทำให้ quote ปิด attribute ก่อนเวลา (`onclick="installPkg("` เท่านั้น) กดปุ่มแล้วได้ `Uncaught SyntaxError: expected expression, got '}'` ทันที — บั๊กนี้มีอยู่เดิมตั้งแต่ก่อนเพิ่มปุ่มถอน แต่ไม่เคยโผล่ให้เห็นเพราะ dev-mode (`VAS_DEV_FAKE_INSTALLED=1`) ทำให้ทุก package โชว์ "Installed" (ปุ่ม disabled ไม่มี onclick) จนกว่าจะรันบนเครื่องจริงที่มี package ยังไม่ติดตั้งแล้วกดปุ่มจริงๆ ครั้งแรก
- แก้เป็น `onclick="fn(\'' + esc(p.id) + '\')"` — ใช้ single quote ครอบค่าที่ฝังใน attribute แทน ไม่ชนกับ double quote ของ attribute เอง

### หน้า "โปรแกรมเพิ่มเติม" — เพิ่มปุ่มถอนการติดตั้ง (uninstall) พร้อม progress bar สำหรับทั้งติดตั้งและถอน
- `src/features/packages/settings.py`: เพิ่ม `uninstall_cmds` ให้ทุก package ใน manifest (git, node, pm2, docker, wireguard, openssh, anydesk, qr-udev) โดยอิงคำสั่งชุดเดียวกับ `LifecycleManager.uninstall_*` ใน `services/reset.py` เพิ่ม `start_uninstall()` / `get_uninstall_queue()` / `is_uninstalling()` คู่กับของเดิมฝั่ง install รวมทั้ง refactor runner เดิมเป็น `_run_commands(cmds, q, stop_on_error)` ใช้ร่วมกันทั้งสองฝั่ง — install หยุดทันทีถ้า command ไหน fail (`stop_on_error=True`), uninstall เป็น best-effort ข้ามไปคำสั่งถัดไปถ้า fail (`stop_on_error=False`) เพราะ `apt purge` ของที่ไม่ได้ติดตั้งอยู่แล้วไม่ควรทำให้ทั้ง flow พัง

- `src/web/templates/apps.html`: เพิ่มปุ่ม "ถอน" (สีแดง, ใช้ `showConfirm()` ตาม convention ของโปรเจกต์ — ห้ามใช้ `window.confirm()`) ให้ package ที่ติดตั้งแล้วและไม่มีตัวอื่นพึ่งพาอยู่ (`can_uninstall`) ถ้ามีตัวอื่นพึ่งพาอยู่จะโชว์ปุ่ม disabled พร้อม tooltip บอกว่าต้องถอนอะไรก่อน — เพิ่ม progress bar (`.pkg-progress-track`/`.pkg-progress-fill`, สไตล์เดียวกับหน้า "อัปเดตระบบ") ในกล่อง modal เดิมที่ใช้ร่วมกันทั้งติดตั้ง/ถอน อัปเดตจาก SSE event `progress` แบบ real-time, เขียวเมื่อเสร็จ (`is-done`) แดงเมื่อ error (`is-error`) — รวม logic ปุ่ม install/uninstall/blocked ทั้งหมดเป็นฟังก์ชันเดียว `actionButtonHtml()` ใช้ร่วมกันทั้งแถว parent และ sub-row (children)

### Database foundation — ย้าย config.json/qr_integrations.json เข้า SQLite + schema migration system + ซ่อม multi-broker
- `src/core/database.py`: แทนที่ `_SCHEMA` เดี่ยวด้วย `_MIGRATIONS: list[tuple[int, str]]` ใช้ `PRAGMA user_version` เก็บเวอร์ชัน schema — version 1 = schema เดิมทั้งหมด (รวม `users` table ที่ย้ายมาจาก `core/auth.py:init_users()`), version 2 = ตารางใหม่ `device_integrations` เพิ่ม `run_migrations()` (apply migration ทีละ version, wrap transaction, ห้าม DROP/DELETE ข้อมูลเดิม), `current_schema_version()`, `latest_schema_version()` — `init_db()` เปลี่ยนเป็น read-only version check (raise `SchemaOutOfDateError` ถ้า DB เก่ากว่าโค้ด) ไม่เขียน schema ตอน boot อีกต่อไป
- Data migration (ผูกกับ migration version 2, insert-then-read-back-verify ก่อนลบไฟล์เสมอ): `config.json` → `mqtt_brokers` (broker ใหม่ `is_primary=1`), `~/.config/vas/qr_integrations.json` → `device_integrations` (device_id="zkteco-qr500") — ถ้าไม่มีไฟล์ต้นทางจะข้ามไม่ error, ลบไฟล์จริงเฉพาะเมื่อ read-back ยืนยันสำเร็จ
- `src/features/mqtt/client.py`: เปลี่ยน singleton `_client` เดี่ยวเป็น `_clients: dict[int, VasMqttClient]` (key=broker_id) รองรับหลาย broker connect พร้อมกันจริง แต่ละตัว enable/disable อิสระต่อกัน เพิ่ม `broker_db_to_config()`, `start_mqtt_broker()`, `stop_mqtt_broker()`, `get_primary_broker_id()` (ใน database.py), `start_all_enabled_brokers()` (แยก try/except ต่อ broker — ตัวหนึ่ง fail ไม่กระทบตัวอื่น) `get_mqtt_client()` เปลี่ยน signature เป็นรับ `broker_id: int | None` (None = primary) `publish_qr_scan()` เปลี่ยนไป publish เข้า broker ที่ `is_primary=1` แทน legacy `_client` (device-aware routing เป็น scope รอบถัดไป) ลบ `load_mqtt_config()`/`save_mqtt_config()` (file-based)
- `src/core/config.py`: ลบ `main_config_path()` (ไม่ใช้ config.json อีกต่อไป)
- `src/features/qr/registry.py`: `load_integrations()`/`save_integrations()` เปลี่ยนไปอ่าน/เขียน `device_integrations` ผ่าน `core.database` แทน `qr_integrations.json` — คง signature เดิม (dict keyed by type) ไม่กระทบ caller ใน `server.py`
- `src/server.py`: `_init_db()` ตอน boot เปลี่ยนเป็น version check — ถ้า schema ไม่ทันสมัย log `[FATAL]` พร้อมบอกให้รัน `vas db migrate` แล้วปฏิเสธ start (fail loud) เปลี่ยน auto-start MQTT ตอน boot จาก single-config เป็น `start_all_enabled_brokers()` แก้ legacy `/api/mqtt/config`, `/api/mqtt/test`, `/api/mqtt/disconnect` และ broker CRUD API (`update`/`delete`/`disconnect`/`test`) ให้ทำงานกับ broker ที่ระบุเท่านั้น (ไม่ทำ global `stop_mqtt()` ตัดทุก broker เหมือนเดิมที่เป็นผลข้างเคียงจาก singleton) sync `MqttConfig.topic` เข้า `mqtt_broker_topics` เพราะ `mqtt_brokers` ไม่มี column `topic` โดยตรง
- `src/cli.py`: เพิ่มคำสั่ง `vas db migrate` (เรียก `run_migrations()` ตรงๆ, dry-run แสดง schema version ปัจจุบัน/ล่าสุดโดยไม่เขียนจริง) `vas install` เรียก migration ต่อท้ายหลัง component อื่นติดตั้งเสร็จ `vas mqtt status/config/test` เปลี่ยนไปอ่าน/เขียน primary broker ใน DB แทน `config.json`
- `src/services/updater.py`: ทั้ง `SelfUpdater.update()` (CLI `vas update`) และ `start_web_update()` (web update) เรียก `run_migrations()` หลัง `shutil.copytree` เสร็จ เพื่อให้ schema ทันสมัยหลังอัปเดตโดยไม่ลบข้อมูลเดิม
- Breaking/known gap ที่ยอมรับตามสโคป: `publish_qr_scan()` ยัง publish ที่ primary broker เท่านั้น (ไม่ device-aware) จนกว่าจะถึงรอบถัดไปตาม TODO `webhook`/`pipe` integration type เก็บ schema ได้แต่ยังไม่มี backend publish logic จริง (ของเดิม ไม่ใช่ regression) `mqtt_broker_form.html` ส่ง `payload_mode: "json"` (ไม่ตรงกับ `PAYLOAD_MODES`) เป็นบั๊กเดิมที่ปล่อยไว้แก้พร้อม UI wiring รอบถัดไปตามที่ระบุใน spec ไม่ใช่ regression จากงานนี้
