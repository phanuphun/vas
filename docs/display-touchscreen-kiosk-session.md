# จอแสดงผล + Touchscreen + Kiosk Mode + Session — คู่มือฉบับอ่านง่าย

> เอกสารนี้เขียนสำหรับคนที่ไม่ได้เชี่ยวชาญ Linux มาก่อน อธิบายว่าระบบตั้งค่าจอ/ทัชสกรีน/โหมด Kiosk ของ VAS ทำงานอย่างไร ทำไมวันนี้ (8 ก.ค. 2569) ถึงเจอปัญหาจอสลับกลับไปหน้า login และ touch เพี้ยน พร้อมสรุปว่าอะไรคือสาเหตุจริงจากโค้ด
>
> ศัพท์เทคนิคทุกคำที่ใช้ในเอกสารนี้ จะมีวงเล็บอธิบายกำกับไว้ให้ในจุดที่ใช้ครั้งแรก

---

## 1. คำศัพท์พื้นฐานที่ต้องรู้ก่อน

| คำศัพท์ | ความหมายแบบง่าย |
|---|---|
| **X11 / Xorg** | โปรแกรมพื้นฐานที่ทำหน้าที่ "วาดหน้าจอ" บน Linux (คล้ายๆ เป็นเอนจินกราฟิกที่ทำให้เกิดหน้าต่าง, เมาส์, การสัมผัสจอ) เครื่อง vending ของเราใช้ตัวนี้เป็นหลัก |
| **Wayland** | โปรแกรมวาดหน้าจอตัวใหม่กว่า X11 ที่ Ubuntu สมัยใหม่ default มาให้ แต่ VAS ปิดไว้ตั้งใจ (`WaylandEnable=false`) เพราะเครื่องมือปรับจอ/ทัช (xrandr/xinput) ทำงานกับ X11 ได้เสถียรกว่า |
| **GDM (GNOME Display Manager)** | หน้าจอ "เลือก user แล้วใส่รหัสผ่าน" ที่ขึ้นตอนเปิดเครื่อง — เป็นโปรแกรมที่ตัดสินใจว่าจะ auto-login เข้า user ไหน และ user นั้นจะได้ session แบบไหน |
| **Session** | การ "ล็อกอินหนึ่งรอบ" ของ user คนหนึ่งบนจอเครื่อง (ตั้งแต่ล็อกอินจนถึงล็อกเอาท์/รีสตาร์ท) แต่ละ session ผูกกับ user คนเดียว |
| **xrandr** | คำสั่งที่ใช้สั่ง "หมุนจอ" (rotate) — ใช้กับหน้าจอ/วิดีโอเอาต์พุตเท่านั้น |
| **xinput** | คำสั่งที่ใช้สั่ง "ปรับทิศทางการสัมผัส" ของทัชสกรีน — คนละคำสั่งกับ xrandr (จอกับทัชเป็นอุปกรณ์คนละตัวในสายตา Linux แม้จะเป็นจอเดียวกันในทางกายภาพ) |
| **Coordinate Transformation Matrix** | ชุดตัวเลข 9 ตัวที่บอกทัชสกรีนว่า "แตะตรงนี้ให้แปลงเป็นพิกัดไหน" — เวลาหมุนจอ ต้องหมุน matrix นี้ให้ตรงกันด้วย ไม่งั้นจอหมุนแต่ touch จะเพี้ยน (แตะบนแต่กลายเป็นล่าง เป็นต้น) |
| **`.xprofile`** | ไฟล์ตั้งค่าที่อยู่ใน home folder ของ user แต่ละคน (เช่น `/home/hapymed/.xprofile`) ที่ Linux จะรันให้อัตโนมัติทุกครั้งที่ **user คนนั้น** ล็อกอินเข้า desktop — เป็นคนละไฟล์กันในแต่ละ user เสมอ |
| **Kiosk Mode** | โหมด "บูตแล้วเปิดเบราว์เซอร์เต็มจอทันที ไม่ต้องมีใครมากดอะไร" เหมาะกับตู้ vending ที่ไม่มีคนคอยดูแลหน้าจอ |
| **Autologin** | การตั้งให้ GDM ล็อกอินเข้า user ที่กำหนดไว้อัตโนมัติโดยไม่ต้องพิมพ์รหัสผ่าน |
| **loginctl** | คำสั่ง Linux ที่ VAS server ใช้ "แอบดู" ว่าตอนนี้มีใครล็อกอินอยู่หน้าจอเครื่องจริงบ้าง (ใช้แทนการเดา) |

## 2. ภาพรวม: มี "3 ระบบ" ที่ดูคล้ายกันแต่เป็นคนละเรื่อง

ปัญหาที่เจอวันนี้ ส่วนใหญ่มาจากการที่ 3 ระบบนี้ทำงาน**แยกจากกัน**แต่ผู้ใช้มองว่าเป็นเรื่องเดียวกัน:

1. **การตั้งค่าจอ/ทัชสกรีน** (หน้า "จอแสดงผล") — หมุนจอ, ปรับทิศทาง touch, ปิดหน้าจออัตโนมัติ, สลับ X11/Wayland
2. **Kiosk Mode** (หน้า "คีออส") — สร้าง user สำหรับ kiosk, ตั้ง autologin, เลือกว่า login แล้วเจอ GNOME หรือ Openbox, ตั้งให้เปิดเบราว์เซอร์อัตโนมัติ
3. **การหาว่า "ใครคือคนที่นั่งอยู่หน้าจอเครื่องตอนนี้"** — กลไกเบื้องหลังที่ทั้ง 2 ระบบข้างบนต้องพึ่งพา เพื่อรู้ว่าจะไปรันคำสั่ง/เขียนไฟล์ที่ home ของใคร

ตัว VAS server เองรันเป็น background service ของทั้งเครื่อง (ไม่ได้ผูกกับ user คนไหนคนหนึ่ง) ดังนั้นเวลาเราสั่ง "หมุนจอ" ผ่านหน้าเว็บ เบื้องหลัง VAS ต้อง**เดา/ตรวจสอบ**ก่อนว่าตอนนี้ควรไปสั่งงานให้ user ไหน — จุดนี้คือรากของปัญหาเกือบทั้งหมดที่เจอวันนี้ (อธิบายละเอียดใน หัวข้อ 5)

## 3. การตั้งค่าจอ/ทัชสกรีน ทำงานอย่างไร (หน้า "จอแสดงผล")

เมื่อกด **Apply** ในหน้า Monitor Setting ระบบจะทำ 3 อย่าง (บางอย่างเลือกได้ว่าจะทำหรือไม่):

### 3.1 Runtime (ทำทันที เห็นผลทันที แต่หายเมื่อรีบูต)
สั่ง `xrandr --rotate` (หมุนจอ) + `xinput set-prop ... Coordinate Transformation Matrix` (ปรับ touch) ทันทีตอนนั้นเลย — ทำทุกครั้งไม่ว่าจะติ๊กตัวเลือกอื่นไว้หรือไม่

### 3.2 Persist Session — toggle "Persist session" (ติ๊กไว้เป็นค่าเริ่มต้น)
เขียนไฟล์ 2 ไฟล์ **ลงใน home folder ของ user คนที่ VAS มองว่ากำลังนั่งอยู่หน้าจอตอนนั้น**:

- `~/.config/vending-auto-setup/display-session.sh` — สคริปต์ที่จะ "รอ" จอ/ทัชปรากฏก่อน (poll ทุก 1 วินาที สูงสุด ~30 ครั้ง หลัง delay เริ่มต้น 5 วินาที) แล้วค่อยสั่งหมุนจอ+ปรับ touch — ที่ต้องรอเพราะหลัง reboot บางที HDMI/ทัชสกรีนยังไม่ปรากฏใน 1-2 วินาทีแรก
- `~/.xprofile` — ถูกแก้ให้เรียกสคริปต์ข้างบนแบบ background ทุกครั้งที่ **user คนนั้น** ล็อกอิน

**จุดสำคัญ:** ไฟล์ทั้ง 2 นี้อยู่ใน home ของ user คนเดียว ถ้า user อื่นล็อกอิน จะไม่เจอไฟล์นี้เลย เพราะ home คนละโฟลเดอร์กัน (ดูหัวข้อ 5 ว่าทำไมถึงเป็นปัญหา)

### 3.3 Persist Xorg — toggle "Persist touch in Xorg" (ปิดไว้เป็นค่าเริ่มต้น)
เขียนไฟล์ **1 ไฟล์เดียวของทั้งเครื่อง** (ไม่ผูกกับ user คนไหน): `/etc/X11/xorg.conf.d/99-vending-touchscreen.conf` — ระบุว่า "ถ้าเจอทัชสกรีนรุ่นนี้ (จับคู่ด้วยชื่อฮาร์ดแวร์) ให้ใช้ matrix นี้เสมอ"

**จุดสำคัญมาก:** ไฟล์นี้ Xorg จะอ่านแค่ **ตอนที่ X server เริ่มทำงานใหม่เท่านั้น** (คือตอน logout แล้ว login ใหม่ หรือ reboot) — กดติ๊กแล้ว Apply แล้วไม่ logout/reboot จะยังไม่เห็นผลอะไรต่างไปเลย ทั้งที่ไฟล์ถูกเขียนจริงแล้ว

**เรื่องที่ทำให้สับสน (ตรงกับที่สังเกตในข้อ 8):** checkbox นี้ถูกออกแบบให้เป็น "ปุ่มสั่งทำตอนนี้" (action) ไม่ใช่ "สถานะที่เคยตั้งไว้" (status) — ดังนั้นทุกครั้งที่ refresh หน้าเว็บ checkbox นี้จะกลับไปไม่ติ๊กเสมอ **โดยตั้งใจ** ไม่ได้แปลว่าการตั้งค่าที่เคยกด Apply ไปก่อนหน้านี้หายไป (ถ้าอยากเช็คว่าไฟล์นี้เคยถูกตั้งไว้จริงไหม ต้องดูที่แท็บ "Config Files" ของหน้าเดียวกัน ซึ่งจะโชว์สถานะไฟล์จริงแยกต่างหาก)

### 3.4 ตัวเลือก "แยกทิศทาง Touch จากจอ" (decouple)
ปกติเวลาหมุนจอ ระบบจะหมุน touch matrix ตามจอไปด้วยอัตโนมัติ (จอ Left → touch ก็ Left) แต่บางเครื่องสายทัชสกรีนต่อกลับด้าน ทำให้ touch ต้องตั้งเป็นคนละทิศกับจอถึงจะตรง — toggle นี้เปิดให้เลือก "ทิศทาง touch" แยกจาก "ทิศทางจอ" ได้อิสระ

VAS จะ**ตรวจจากเครื่องจริง** (อ่านค่า Coordinate Transformation Matrix ปัจจุบันของทัชสกรีนผ่าน `xinput list-props`) แล้วเปิด toggle นี้ให้อัตโนมัติทุกครั้งที่โหลดหน้า ถ้า detect ได้ว่ามีการตั้ง touch rotation อยู่จริง (ไม่ว่าจะตรงหรือไม่ตรงกับทิศทางจอ) — ก่อนหน้านี้เคยมีบั๊กที่ระบบเปิด toggle นี้ให้เฉพาะตอนที่ touch "ไม่ตรง" กับจอเท่านั้น ทำให้เคสที่ตั้งไว้ตรงกันพอดี (เช่นทั้งคู่ Left) ดูเหมือน toggle ปิดเองทุกครั้งที่ refresh — แก้ไปแล้วในคอมมิทของวันนี้ (ดูหัวข้อ 6)

### 3.5 ปุ่ม "Reset" — รีเซ็ตกลับ normal
ทำ 4 อย่างตามลำดับ: (1) สั่งจอ+ทัชกลับเป็น normal ทันที (2) ลบ block ที่แทรกไว้ใน `.xprofile` (3) ลบไฟล์สคริปต์ `display-session.sh` (4) ลบไฟล์ Xorg touchscreen config — ทำให้เครื่องกลับไปเป็นสถานะ "ไม่มี VAS ตั้งค่าอะไรเลย" ทุกที่ในคราวเดียว

## 4. Kiosk Mode ทำงานอย่างไร (หน้า "คีออส")

Kiosk Mode ประกอบด้วย 4 ส่วนที่แยกไฟล์กันชัดเจน:

1. **สร้าง Linux user แยก** (เช่น `kiosk-user`) — เพื่อไม่ให้ตู้ vending ใช้สิทธิ์ของ user แอดมิน (เช่น `hapymed`) ตรงๆ
2. **GDM auto-login** — เขียนที่ `/etc/gdm3/custom.conf` (ไฟล์เดียวกับที่คุม Wayland on/off ในหัวข้อ 3 แต่คนละ key — `AutomaticLoginEnable`/`AutomaticLogin` ไม่ชนกับ `WaylandEnable`) กำหนดว่า "บูตเครื่องแล้วให้ล็อกอินเข้า user ไหนอัตโนมัติ"
3. **Session type ต่อ user** — เขียนที่ `/var/lib/AccountsService/users/<username>` บอกว่า user คนนี้ล็อกอินแล้วจะเจอ GNOME (ปกติ) หรือ Openbox (เบาและเรียบง่ายกว่า เหมาะกับตู้ vending) — คนละไฟล์กับข้อ 2 (ข้อ 2 ตอบว่า "จะ login ให้ไหม", ข้อ 3 ตอบว่า "login แล้วเจออะไร")
4. **Autostart เบราว์เซอร์** — สคริปต์ที่เปิด Chromium แบบเต็มจอทันทีหลัง login (GNOME ใช้ `~/.config/autostart/*.desktop`, Openbox ใช้ `~/.config/openbox/autostart`)

ทั้ง 4 ส่วนนี้ **ไม่เกี่ยวกับการตั้งค่าจอ/ทัชสกรีนในหัวข้อ 3 โดยตรง** — เป็นคนละระบบ คนละไฟล์ทั้งหมด แต่เชื่อมกันทางอ้อมผ่านคำถามเดียวกันคือ "ตอนนี้ home ไหนที่ VAS กำลังเขียนถึง" (หัวข้อ 5)

## 5. หัวใจของปัญหา: VAS หา "คนที่นั่งอยู่หน้าจอ" ยังไง และทำไมมันพลาดได้

VAS server รันเป็น systemd service ด้วยสิทธิ์ root ตลอดเวลา ไม่ได้ผูกกับ user คนไหนเป็นการถาวร ดังนั้นทุกครั้งที่ต้อง (ก) สั่ง xrandr/xinput ผ่านหน้าเว็บ หรือ (ข) เขียนไฟล์ `.xprofile`/`display-session.sh` ลง home folder — VAS ต้องมีวิธี "หาตัวจริง" ว่าใครคือ user ที่ล็อกอินอยู่หน้าจอเครื่องตอนนั้น

วิธีที่ใช้อยู่ตอนนี้ (แก้ไปแล้ววันนี้ในคอมมิท `b867b33`/`4f7f2a8` — ดูหัวข้อ 6) คือใช้คำสั่ง `loginctl` ของ Linux เอง ไล่ดูรายชื่อ session ที่กำลังล็อกอินอยู่ แล้วเลือก session ที่ดูน่าเชื่อถือที่สุดว่าเป็น "คนที่นั่งอยู่หน้าจอเครื่องจริง" (มี "Seat" ติดอยู่ — Seat แปลว่า "มีจอ/คีย์บอร์ด/เมาส์ต่ออยู่จริงกับเครื่องนี้" ต่างจาก session ที่ต่อผ่าน SSH ทางไกลซึ่งไม่มี Seat)

**ปัญหาคือ**: ค่านี้เปลี่ยนไปตามว่า **ใครล็อกอินอยู่ ณ วินาทีนั้นจริงๆ** ไม่ใช่ "user ที่ตั้ง kiosk ไว้" เสมอไป — ผลกระทบที่เจอวันนี้ทั้งหมดมาจากจุดนี้:

- ถ้าตอนที่กด Apply ตั้งค่าจอ/ทัช คุณล็อกอินด้วย `hapymed` อยู่ (เช่นเข้าไปทดสอบ/ปรับผ่านหน้าเว็บโดยตัวเองนั่งอยู่หน้าเครื่อง) การตั้งค่าจะถูกเขียนลง **`/home/hapymed/.xprofile`**
- พอเปิด Kiosk Mode แล้วตั้งให้ auto-login เข้า `kiosk-user` แทน — `kiosk-user` มี home ของตัวเองคือ `/home/kiosk-user` ซึ่ง**ไม่มีไฟล์ที่เพิ่งตั้งไว้เลย** เพราะไฟล์ถูกเขียนไปอีก home หนึ่ง
- ผลคือ: ทุกครั้งที่ `kiosk-user` ล็อกอิน จะไม่มีสคริปต์ไหนมาสั่งหมุนจอ/ปรับ touch ให้เลย → จอจึงกลับไปที่ค่าเริ่มต้นของระบบเสมอ ("normal") — ตรงกับที่สังเกตในข้อ 4 ของรายงานพอดี
- ในทางกลับกัน การเปิดหน้า "จอแสดงผล" เพื่อดู HDMI/ทัชสกรีนที่ detect ได้ ก็ใช้กลไกเดียวกันนี้ในการเลือกว่าจะ "แอบส่องจอของใคร" ผ่าน `xrandr`/`xinput` — ถ้าตอนนั้น `loginctl` ยังไม่เห็น session ของ `kiosk-user` ชัดเจนพอ (เช่น session เพิ่ง auto-login มาไม่กี่วินาที ยังไม่ถูก systemd ติดป้าย Seat/ประเภท session ครบ) หน้าเว็บจะมองไม่เห็น HDMI/ทัชสกรีนเลย จนกว่าจะมี session ที่ `loginctl` มั่นใจกว่า (เช่น `hapymed` login เข้ามาจริง) ปรากฏขึ้นมาแทน — ตรงกับข้อ 3 ของรายงาน

พูดง่ายๆ คือ **"ตั้งค่าจอตอนไหน ก็ไปอยู่ในบ้านของคนที่ล็อกอินอยู่ตอนนั้น"** ถ้าอยากให้ `kiosk-user` มีค่าที่ถูกต้องติดตัวไปด้วย ต้องกด Apply **ตอนที่ `kiosk-user` เป็นคนล็อกอินอยู่หน้าจอเครื่องจริง** (ไม่ใช่ตอนที่ `hapymed` ล็อกอินอยู่แล้วปรับผ่านเว็บจากระยะไกล)

ส่วนไฟล์ Xorg (`/etc/X11/xorg.conf.d/99-vending-touchscreen.conf` ในหัวข้อ 3.3) ไม่ผูกกับ user คนไหนเลย เขียนครั้งเดียวใช้ได้กับทุก user บนเครื่อง — **นี่คือทางที่ปลอดภัยกว่าเวลาจะสลับไปมาระหว่าง user หลายคน (hapymed/kiosk-user)** เพราะไม่ต้องกังวลว่า "ใครล็อกอินอยู่ตอนกด Apply" แต่แลกมาด้วยข้อเสียคือต้อง logout/reboot ถึงจะเห็นผล (ตามหัวข้อ 3.3)

## 7. ไล่ทีละข้อ: สรุปสาเหตุของพฤติกรรมที่เจอวันนี้

**ข้อ 1-2 (จอกลับไปเป็น login, ใช้คอมมิทเช้าตั้ง touch แล้ว restart กลับมาปกติ):** ตอนนั้นยังไม่ได้เปิด Kiosk Mode เลย เครื่องน่าจะ auto-login เข้า user เดิม (เช่น hapymed) ตามปกติ การตั้งค่าจอ/ทัชจึงเขียนลง home เดียวที่ใช้งานจริงตลอด ไม่มีปัญหาเรื่อง "คนละ home" เกิดขึ้น — สอดคล้องกับที่ตั้งค่าได้ปกติ

**ข้อ 3 (ตั้ง Kiosk ด้วย kiosk-user แล้วไม่เจอ HDMI จนกว่าจะ login hapymed):** ตรงกับหัวข้อ 5 — ตอนที่หน้าเว็บพยายามหา HDMI/ทัชสกรีนผ่าน `loginctl`, session ของ `kiosk-user` อาจยังไม่ถูกระบบมองว่าเป็น "session หน้าจอเครื่องจริง" ชัดเจนพอ (โดยเฉพาะช่วงแรกหลัง auto-login หรือถ้า Openbox ไม่ถูกติดป้าย session type แบบเดียวกับ GNOME) พอ hapymed login เข้ามาจริง `loginctl` เจอ session ที่ชัดเจนกว่า จึงกลับมาเห็น HDMI ปกติ

**ข้อ 4 (login kiosk-user แล้วจอ default เป็น normal เสมอ):** นี่คือจุดสำคัญที่สุด — น่าจะเป็นเพราะการตั้งค่าจอ/ทัชที่ทำไว้ก่อนหน้า (ตอนเปิด Kiosk Mode ครั้งแรก) ถูกกด Apply ตอนที่ user อื่น (ไม่ใช่ kiosk-user) เป็นคนล็อกอินอยู่ ทำให้ `.xprofile`/`display-session.sh` ไปอยู่ผิด home — kiosk-user จึงไม่มีสคริปต์ตั้งค่าจอให้เรียกใช้เลยทุกครั้งที่ล็อกอิน (ดูหัวข้อ 5)

**ข้อ 6 (kiosk เห็น HDMI ได้แล้วหลังแก้):** สอดคล้องกับคอมมิท `b867b33`/`4f7f2a8` (เปลี่ยนมาถาม loginctl + ปรับ logic Seat) ที่ทำให้ detect session ของ kiosk-user ได้แม่นขึ้น

**ข้อ 7 (ทดลอง set touch เป็น left/right/normal สลับไปมา จน touch ตรง แต่พอ toggle Persist Xorg แล้ว restart กลับเพี้ยนอีก):**

- ที่ต้องกลับไปเปิด toggle "แยกทิศทาง Touch จากจอ" แล้วเลือก normal/left/right เองอีกรอบ เพราะอย่างที่อธิบายในหัวข้อ 3.4 — ถ้า toggle นี้ปิดอยู่ ระบบจะบังคับให้ touch หมุนตามจอเสมอ ต่อให้ hardware จริงต้องการทิศทางอื่น

**ข้อ 8 (toggle Persist touch in Xorg ปิดเองทุกครั้งที่ refresh ทั้งที่เพิ่ง toggle — สงสัยว่ามี DB เก็บค่าไหม):**
ตอบตรงๆ คือ **ไม่มี** — toggle นี้ไม่ได้เก็บ/อ่านจากฐานข้อมูลใดๆ เลย (ยืนยันจาก `docs/database.md`: ตาราง Database ของ VAS มีแค่ `users`, `qr_scans`, `mqtt_events`, `audit_log`, `config_history`, `mqtt_brokers`, `mqtt_broker_topics`, `device_integrations` — ไม่มีตารางเก็บค่าจอ/ทัช/kiosk เลยสักตาราง) การตั้งค่าจอ/ทัช/kiosk ทั้งหมดเก็บเป็น**ไฟล์ในระบบปฏิบัติการโดยตรง** (`.xprofile`, `display-session.sh`, ไฟล์ Xorg, `/etc/gdm3/custom.conf`, `/var/lib/AccountsService/...`) หน้าเว็บอ่านค่ามาแสดงสดๆ จากไฟล์เหล่านี้ทุกครั้งที่โหลดหน้า ไม่ได้ cache ไว้ที่ไหน

ส่วน toggle "Persist touch in Xorg" ที่ดูเหมือนปิดเองทุกครั้ง — เป็นความตั้งใจของผู้พัฒนา (ยืนยันใน CHANGELOG.md): ออกแบบให้เป็น "ปุ่มสั่งเขียนไฟล์รอบนี้" ไม่ใช่ "ตัวบอกสถานะว่าเคยตั้งไว้หรือยัง" เพราะมีที่แสดงสถานะไฟล์จริงแยกไว้ต่างหากแล้วในแท็บ Config Files ของหน้าเดียวกัน — จึงไม่ใช่บั๊ก แต่เป็น UI ที่สื่อสารความหมายไม่ชัดเจนพอ ทำให้เข้าใจผิดว่าค่าหาย

## 9. คำสั่งตรวจสอบไฟล์ของแต่ละ user (รันบนเครื่อง Ubuntu จริง ผ่าน SSH หรือ terminal)

รันคำสั่งพวกนี้ด้วย `sudo` เสมอ (ต้องมีสิทธิ์ root ถึงจะอ่านไฟล์ใน home ของ user คนอื่นและไฟล์ใน `/etc` ได้ครบ)

### 9.1 ใช้คำสั่งสำเร็จรูปของ VAS ก่อน (เร็วที่สุด)

```bash
sudo vas check           # ภาพรวมทั้งเครื่อง: session ปัจจุบัน, GDM Wayland, config .xprofile/script/Xorg touchscreen ของ "user ที่ VAS มองว่าล็อกอินอยู่ตอนนี้"
sudo vas kiosk status    # user ที่มี, ใครถูก auto-login, session type (gnome/openbox), autostart พร้อมหรือยัง
sudo vas display status  # xrandr + xinput ของ session ปัจจุบัน (จอ/ทัชที่ detect ได้ตอนนี้)
sudo vas display list-touch   # รายชื่อทัชสกรีนที่ detect ได้ พร้อม xinput ID
```

### 9.2 สคริปต์ไล่ตรวจทุก user ที่มี home (เจาะไฟล์ตั้งค่าจอ/ทัชของแต่ละคน)

Copy ทั้งก้อนนี้รันได้เลย จะไล่ทุก user จริงในเครื่อง (ไม่รวม system account) แล้วโชว์ว่าใครมีไฟล์อะไรบ้าง:

```bash
#!/usr/bin/env bash
echo "=== GDM autologin / Wayland (/etc/gdm3/custom.conf) ==="
sudo cat /etc/gdm3/custom.conf 2>/dev/null || echo "(ไม่พบไฟล์)"

echo
echo "=== Xorg touchscreen calibration ระดับเครื่อง (ใช้ร่วมกันทุก user) ==="
sudo cat /etc/X11/xorg.conf.d/99-vending-touchscreen.conf 2>/dev/null || echo "(ยังไม่ได้ตั้งค่า / ไม่พบไฟล์)"

echo
echo "=== ไล่ทุก user (UID 1000-60000) ==="
getent passwd | awk -F: '$3>=1000 && $3<60000 {print $1":"$6}' | while IFS=: read -r user home; do
  echo "----------------------------------------"
  echo "USER: $user   HOME: $home"

  echo "  [AccountsService session type]"
  sudo cat "/var/lib/AccountsService/users/$user" 2>/dev/null | sed 's/^/    /' || echo "    (ไม่พบไฟล์ — ยังไม่เคยตั้ง session type, default=gnome)"

  echo "  [.xprofile มี VAS block ไหม]"
  if sudo test -f "$home/.xprofile"; then
    sudo grep -q "vending-auto-config: display-session" "$home/.xprofile" \
      && echo "    มี — จะเรียก display-session.sh ตอน login" \
      || echo "    มีไฟล์ .xprofile แต่ไม่มี block ของ VAS"
  else
    echo "    ไม่มีไฟล์ .xprofile เลย"
  fi

  echo "  [display-session.sh]"
  script="$home/.config/vending-auto-setup/display-session.sh"
  if sudo test -f "$script"; then
    perm=$(sudo stat -c '%A' "$script")
    echo "    มีไฟล์ ($perm)"
    sudo grep -E '^(OUTPUT|TOUCH_DEVICE|ROTATE|TOUCH_ROTATE|MATRIX)=' "$script" | sed 's/^/    /'
  else
    echo "    ไม่มีไฟล์ — user นี้จะไม่ถูกหมุนจอ/ปรับ touch อัตโนมัติตอน login เลย"
  fi

  echo "  [display-session.log ล่าสุด]"
  log="$home/.config/vending-auto-setup/display-session.log"
  sudo test -f "$log" && sudo tail -n 8 "$log" | sed 's/^/    /' || echo "    (ยังไม่เคยรัน หรือไม่มี log)"

  echo "  [Kiosk autostart]"
  sudo test -f "$home/.config/autostart/vas-kiosk.desktop" && echo "    GNOME autostart: มี"
  sudo test -f "$home/.config/openbox/autostart" && echo "    Openbox autostart: มี"
done
```

### 9.3 เช็คสถานะ "ตอนนี้จอ/ทัชถืออะไรอยู่จริง" (live state)

ต้องรันตอนที่มี session กราฟิกกำลังเปิดอยู่ (ไม่ใช่ผ่าน SSH เฉยๆ เพราะ `xrandr`/`xinput` ต้องมี `DISPLAY` ให้เชื่อมกับ X session จริง):

# ดูว่า loginctl เห็น session ไหนอยู่บ้าง คอลัมน์ไหนมี "Seat" = คนที่นั่งอยู่หน้าเครื่องจริง

## 10. ทำไม Touch Screen ยังเพี้ยนได้ ทั้งที่ไฟล์ `99-vending-touchscreen.conf` ไม่สนใจว่าเป็น user ไหน

คำถามนี้ตรงประเด็นมาก — คำตอบคือไฟล์ Xorg นั้น **"ไม่สนใจ user"** จริง แต่มันไม่ใช่เจ้าเดียวที่คุม touch matrix ยังมีอีก 2 จุดที่แก้ไขค่าเดียวกันนี้ทับได้ตลอดเวลา และ 2 จุดนั้น**สนใจ user**:

**1) มันคุมแค่ "ค่าเริ่มต้นตอนจอเริ่มทำงาน" ไม่ใช่ "ค่าที่บังคับตลอดไป"**

`Coordinate Transformation Matrix` เป็น property ที่แก้ **สดๆ ตอนไหนก็ได้** ด้วยคำสั่ง `xinput set-prop` — ไฟล์ Xorg แค่กำหนดว่า "ตอนทัชสกรีนถูกมองเห็นครั้งแรกตอน X server เริ่มทำงาน ให้ตั้งค่าเริ่มต้นเป็นอะไร" หลังจากนั้นถ้ามีอะไรมาสั่ง `xinput set-prop` ซ้ำ (ไม่ว่าจะจาก script ไหน หรือกดปุ่มในหน้าเว็บ) ค่าล่าสุดจะชนะเสมอ ไม่สนใจว่าไฟล์ Xorg เคยตั้งไว้เป็นอะไร

**2) สคริปต์ต่อ user (`display-session.sh`) รันทีหลังเสมอ และมันสนใจ user**

ลำดับตอน login จริงคือ: X server เริ่ม (อ่านไฟล์ Xorg → ตั้ง matrix เริ่มต้น) **ก่อน** แล้ว session ของ user ถึงเริ่ม แล้ว `.xprofile` ถึงเรียก `display-session.sh` (ของ **home คนนั้นคนเดียว**) มา "เขียนทับ" matrix อีกที ด้วยค่าที่บันทึกไว้ตอน Apply ครั้งล่าสุดของ **user คนนั้น** — ถ้า `hapymed` เคย Apply ไว้เป็น "left" และ `kiosk-user` ไม่เคย Apply เลย (หรือ Apply ไว้ตอนที่ decouple ปิดอยู่พอดี) สคริปต์ของ 2 คนนี้จะเขียนทับค่าจากไฟล์ Xorg เป็นคนละค่ากัน **ทั้งที่ไฟล์ Xorg เขียนไว้ค่าเดียวก็ตาม** — นี่คือจุดที่ "user คนไหนก็มีผล" แอบกลับเข้ามาแม้ Xorg conf จะ user-agnostic ก็ตาม

**3) ไม่มีไฟล์ระดับเครื่องสำหรับ "ทิศทางการหมุนจอ" เลย — มีแต่ตัวหมุน touch เท่านั้น**

จุดนี้สำคัญที่สุด: VAS มีกลไก persist ระดับเครื่อง (ไม่ผูก user) ให้แค่ **touch calibration** เท่านั้น (ไฟล์ `99-vending-touchscreen.conf`) ส่วน **การหมุนจอ** (`xrandr --rotate`) **ไม่มี** ไฟล์ระดับเครื่องแบบนี้เลย — มีแค่ 2 ทาง: (ก) กดปุ่ม Apply สดๆ ตอนนั้น หรือ (ข) สคริปต์ต่อ user ที่ต้องรอ login เสร็จก่อนถึงจะรัน

ผลคือ **หน้าจอ login ของ GDM เอง** (ก่อนที่ใครจะ login เข้า user ไหนเลย) จะไม่มีอะไรมาสั่งหมุนจอให้ทั้งสิ้น จอจะอยู่ในทิศทาง native/default ของฮาร์ดแวร์เสมอ **ในขณะที่ touch matrix ที่มาจากไฟล์ Xorg (ถ้าตั้งไว้เป็น left/right ตาม physical mounting ของจอ) ยังถูกใช้งานอยู่** เพราะ Xorg โหลดไฟล์นั้นตั้งแต่ก่อน login แล้ว — จึงเกิดอาการ "จอกลับไปเป็นหน้า login แล้ว touch เพี้ยน" ได้ทันที (ตรงกับข้อ 1 ในรายงานเป๊ะๆ) เพราะหน้าจอ login ไม่ได้หมุนตาม แต่ touch ที่ผูกกับฮาร์ดแวร์ถูกคาลิเบรตไว้สำหรับจอที่หมุนแล้ว

**4) `MatchProduct` ต้องตรงชื่อฮาร์ดแวร์เป๊ะ**

ไฟล์ Xorg จับคู่ด้วยชื่อทัชสกรีน (`MatchProduct "ชื่ออุปกรณ์"`) ถ้าชื่อที่ detect ได้ตอน boot ต่างไปจากตอนที่เขียนไฟล์ไว้แม้เล็กน้อย (เช่น driver คนละตัว, ต่อผ่าน hub คนละพอร์ต) กฎนี้จะไม่ match เลย แล้ว Xorg จะปล่อยให้ touch เป็นค่า identity เริ่มต้น — ควรเช็คด้วยคำสั่งในข้อ 9.3 (`xinput list-props ... | grep Coordinate...`) เทียบกับชื่อใน `MatchProduct` ของไฟล์ 99-... ว่าตรงกันจริงหรือไม่ทุกครั้งที่สงสัย

**สรุป:** ไฟล์ Xorg ตอบโจทย์แค่ "ตั้งค่าเริ่มต้นให้ตรงกันทุก user" แต่ไม่ได้การันตีว่า "จะไม่มีอะไรมาเขียนทับทีหลัง" และไม่ได้แก้ปัญหาเรื่องการหมุนจอเลย (คนละกลไกกัน) — ถ้าจะให้ชัวร์สุดว่า touch จะตรงตลอดไม่ว่าใคร login ต้องเช็คให้ครบทั้ง 3 จุดพร้อมกัน: (1) ไฟล์ Xorg ตรงกับชื่อฮาร์ดแวร์จริง (2) สคริปต์ต่อ user ของทุก user ที่จะ login ถูก Apply ด้วยค่าที่ตรงกัน ไม่ใช่แค่ user เดียว (3) ยอมรับว่าหน้าจอ login เองจะไม่หมุนตาม ถ้าฮาร์ดแวร์จอไม่ auto-rotate เอง

## 12. Runbook: แยก `kiosk-user` ออกจาก `hapymed` ให้เป็นอิสระจากกันจริงๆ

เช็คจริงบนเครื่องพบว่า ณ ตอนนั้น GDM auto-login ชี้ไปที่ `hapymed` ตรงๆ (ไม่ใช่ `kiosk-user`), session type เป็น `gnome`, autostart chromium ก็ผูกกับ `/home/hapymed` — `kiosk-user` มีอยู่ (สร้างโดย VAS) แต่ไม่ได้ถูกใช้งานเป็น kiosk account เลย ทำให้ config ทั้งหมด (auto-login/session-type/autostart/การตั้งค่าจอ-ทัช) ไปกองอยู่ที่บ้านเดียว (hapymed) — ถ้าต้องการแยกจริง ให้ทำตามลำดับนี้บนเครื่อง (ผ่าน SSH หรือ VPN ก็ได้ทั้งหมด ยกเว้นขั้นที่ระบุไว้ชัดเจน)

แบ่งเป็น 3 ช่วง: **ช่วงที่ 1 เตรียม kiosk-user** (ทำตอน hapymed ล็อกอินอยู่ได้ ปลอดภัย เพราะระบุ username ตรงๆ ในทุกคำสั่ง) → **ช่วงที่ 2 สลับจริง** (reboot) → **ช่วงที่ 3 ตั้งจอ/ทัช** (ต้องรอให้ช่วงที่ 2 ยืนยันเสร็จก่อนเท่านั้น)

**ช่วงที่ 1 — เตรียม kiosk-user**

```bash
# 1. เคลียร์ของเดิมออกจาก hapymed ก่อน (ปิด autologin ชั่วคราว + ลบไฟล์ autostart chromium ของ hapymed)
sudo vas kiosk stop --username hapymed

# 2. ตั้ง session type ให้ kiosk-user (openbox เบากว่า, หรือใช้ gnome ถ้าต้องการคงพฤติกรรมเดิม)
sudo vas kiosk session-type --username kiosk-user --type openbox

# 3. ตั้ง autostart เบราว์เซอร์ให้ kiosk-user (ใช้ URL เดิมที่เคยตั้งไว้)
sudo vas kiosk autostart --username kiosk-user --session-type openbox \
  --url https://vending.hapysterile.xenex.io/ --restart-delay 2

# 4. สลับ auto-login ไปที่ kiosk-user
sudo vas kiosk autologin --username kiosk-user --enable

# 5. ตรวจสอบก่อน reboot — ต้องเห็น kiosk-user [auto-login], Session Type: openbox, Readiness 4/4
sudo vas kiosk status
```

**ช่วงที่ 2 — สลับจริง**

```bash
# 6. reboot เครื่อง — GDM auto-login เข้า kiosk-user เอง ไม่ต้องพิมพ์รหัสผ่าน
sudo reboot
```

```bash
# 7. เช็ค (หลังบูตเสร็จ) ว่า VAS มองเห็น kiosk-user เป็น active session จริง — ต้องทำ "ก่อน" ไปข้อ 8 เสมอ
sudo vas check
# ดูบรรทัด [Session] ต้องเป็น session ของ kiosk-user แล้ว
# ถ้ายังโชว์ hapymed แปลว่ายังไม่สลับจริง (เช่น hapymed ยัง login ค้างอยู่ทางกายภาพ) ห้ามไปข้อ 8
```

**ช่วงที่ 3 — ตั้งจอ/ทัชให้ kiosk-user (ทำได้เฉพาะหลังข้อ 7 ยืนยันแล้วเท่านั้น)**

```bash
# 8. เข้าเว็บ "จอแสดงผล" (ผ่าน VPN ก็ได้ ปลอดภัยแล้วตอนนี้เพราะ kiosk-user เป็น active session)
#    ปรับหมุนจอ + ทิศทาง touch -> ติ๊ก Persist session (default) + Persist touch in Xorg -> กด Apply

# 9. ยืนยันว่า config ไปลงบ้านถูกคนแล้ว
sudo vas check
# [Display Config] ต้องขึ้น /home/kiosk-user/.xprofile และ /home/kiosk-user/.config/vending-auto-setup/display-session.sh

sudo grep -E '^(OUTPUT|TOUCH_DEVICE|ROTATE|TOUCH_ROTATE|MATRIX)=' \
  /home/kiosk-user/.config/vending-auto-setup/display-session.sh

# 10. reboot อีกรอบ ทดสอบทั้งกระบวนการตั้งแต่ boot จริง
sudo reboot
```

หลังทำครบ: `hapymed` จะกลับไปเป็นบัญชีแอดมินธรรมดา (เลือก login เองจาก GDM greeter ได้ ไม่ auto-login ไม่มี chromium เด้ง) ส่วน `kiosk-user` จะมี auto-login/session-type/autostart/การตั้งค่าจอ-ทัชเป็นของตัวเองครบ แยกจาก hapymed อย่างสมบูรณ์

**สำคัญที่สุด — ห้ามข้ามลำดับ**: ต้องทำช่วงที่ 1 ให้จบ + reboot (ช่วงที่ 2) + ยืนยันข้อ 7 ว่า kiosk-user เป็น active session จริงแล้ว **ก่อน** ถึงจะไปทำช่วงที่ 3 (Apply จอ/ทัช) ได้ เพราะปุ่ม Apply หน้า "จอแสดงผล" ไม่มีช่องเลือก user เป้าหมายเลย มันเขียนลง home ของ **user ที่ loginctl เห็นว่ากำลังล็อกอินอยู่หน้าจอเครื่องจริง ณ ขณะนั้นเท่านั้น** ไม่เกี่ยวกับว่าคุณสั่งงานผ่านเบราว์เซอร์จากเครื่องไหน (นั่งอยู่หน้าเครื่อง vending เอง หรือต่อ VPN เข้า IP จากระยะไกล ผลเหมือนกันทุกประการ เพราะ backend ที่รันคำสั่งจริงอยู่ที่เครื่อง vending เสมอ) — ถ้า hapymed ยังเป็น active session อยู่ตอนกด Apply ค่าจะไปลง `/home/hapymed` ซ้ำ ทั้งที่เพิ่งสลับ auto-login ไป kiosk-user แล้วก็ตาม แล้ว kiosk-user ก็จะยังไม่มีสคริปต์ตั้งค่าจอ/ทัชติดตัวไปด้วยอยู่ดี

ตรงนี้ต่างจากหน้า "คีออส" (ขั้นที่ 1-5 ในช่วงที่ 1) ซึ่งใช้ username ที่เลือกในฟอร์มตรงๆ (ไม่พึ่ง loginctl) จึงสั่งจากที่ไหนก็ได้ผลถูกต้องเสมอไม่ว่าจะสั่งตอนใครล็อกอินอยู่ก็ตาม — ความไม่สมมาตรนี้ (Kiosk page = user-explicit ปลอดภัย, Display page = session-implicit ต้องระวัง) เป็นข้อจำกัดของดีไซน์ปัจจุบันที่ยังไม่มีช่องให้เลือก "ตั้งค่าให้ user ไหน" ตรงๆ ในหน้า Display — ถ้าต้องการแก้ที่ต้นตอถาวร ควรเพิ่ม dropdown เลือก target user ในหน้า Display เหมือนหน้า Kiosk (เป็นการแก้โค้ดแยกต่างหาก ยังไม่ได้ทำในรอบนี้)

## 13. รหัสผ่านของ `kiosk-user`

`kiosk-user` ที่ VAS สร้างให้ (`useradd -m -c "VAS Kiosk User" -s /bin/bash kiosk-user` ใน `src/features/kiosk/manager.py`) **ไม่มีการตั้งรหัสผ่านให้เลย** — ไม่มีขั้นตอน `passwd`/`chpasswd` อยู่ในโค้ดทั้งฝั่ง CLI และหน้าเว็บ (ทั้งตอนสร้าง user ผ่าน `vas kiosk create-user` และผ่าน `POST /api/kiosk/users`) ผลคือบัญชีนี้ถูกล็อกรหัสผ่านไว้ตามค่า default ของ `useradd` บน Ubuntu (`!` ใน `/etc/shadow`) — พิมพ์อะไรก็ login ด้วยรหัสผ่านไม่ผ่านแน่นอน

**ไม่จำเป็นต้องตั้งรหัสผ่านเลยถ้าใช้ผ่าน GDM auto-login เท่านั้น** เพราะ auto-login ข้ามการเช็ครหัสผ่านไปเลยโดยดีไซน์ปกติของ GDM — ตรงกับการใช้งานจริงที่ kiosk-user ไม่ได้ต้องการให้ใครพิมพ์รหัสผ่าน login เองอยู่แล้ว

ต้องตั้งรหัสผ่านเฉพาะกรณีต้องการอย่างใดอย่างหนึ่งนี้:

```bash
sudo passwd kiosk-user   # ตั้งรหัส — ใช้เมื่อจะเลือก kiosk-user เองจากหน้า GDM greeter (ไม่ผ่าน auto-login) หรือ SSH เข้าตรงๆ ด้วย username/password
```

หรือถ้าแค่ต้องการเปิด shell เข้าไปดู/แก้ไฟล์ใน home ของ kiosk-user (เช่นตามคำสั่งหัวข้อ 9) โดยไม่ต้องตั้งรหัสผ่านเลย ใช้สิทธิ์ sudo ของ admin (เช่น hapymed) สลับ user แทนได้:

```bash
sudo su - kiosk-user   # เปิด shell ธรรมดา ใช้เช็ค/แก้ไฟล์ได้ แต่ไม่ใช่ session กราฟิกจริง ใช้แทนการ login กราฟิกไม่ได้
```
